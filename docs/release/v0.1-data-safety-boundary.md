# CreoNow v0.1 — 数据安全边界声明

> **版本**: v0.1 · **发布日期**: 2026-03-09 · **状态**: 正式

---

## 一、声明目标

本文档如实说明 CreoNow v0.1 对用户数据的存储方式、加密状态与安全边界。

CreoNow 是一款本地优先的 AI 创作 IDE。所有创作内容存储于用户本地计算机，应用层不引入额外加密，安全性依赖于操作系统级保护。本文档旨在让用户对数据安全状况有明确、诚实的预期。

---

## 二、数据存储总表

| 数据类型 | 存储位置 | 存储格式 | 加密状态 | 访问控制 |
|----------|---------|---------|---------|---------|
| 项目文件（content, metadata） | `{userData}/data/creonow.db` | 📄 明文 SQLite（`better-sqlite3`，WAL 模式） | 📄 无应用级加密 | OS 文件系统权限（NTFS / APFS） |
| 知识图谱（KG entities, relations） | `{userData}/data/creonow.db` | 📄 明文 SQLite（`kg_entities`, `kg_relations` 表） | 📄 无应用级加密 | OS 文件系统权限 |
| Memory / Episode（含向量索引） | `{userData}/data/creonow.db` | 📄 明文 SQLite（`memory_episodes` 表 + `user_memory_vec` 虚拟表，`sqlite-vec` 扩展） | 📄 无应用级加密 | OS 文件系统权限 |
| API Keys（敏感凭证） | `{userData}/data/creonow.db`（加密后写入 `settings` 表） | 🔐 Electron `safeStorage` 加密（前缀 `__safe_storage_v1__:`，Base64 编码） | 🔐 已加密 | Electron safeStorage + OS 系统 Keychain |

### 存储路径说明

- `{userData}` = Electron `app.getPath("userData")`，OS 对应位置：
  - **Windows**: `%APPDATA%\CreoNow`
  - **macOS**: `~/Library/Application Support/CreoNow`
  - **Linux**: `~/.config/CreoNow`
- 数据库文件：`{userData}/data/creonow.db`（`db/paths.ts` → `getDbPaths()`）
- 数据库初始化：`db/init.ts` → `initDb()` — `new Database(dbPath)` + WAL 模式 + `foreign_keys = ON`

## 三、三个明确的"不"

### 3.1 不加密创作内容

CreoNow **不**对项目文件、知识图谱、Memory 等创作内容施加应用级加密。

所有创作数据以明文形式存储在本地 SQLite 数据库中。这是有意为之的设计决策：

- **可调试性**：明文存储允许用户使用标准 SQLite 工具直接查看和恢复数据
- **性能优先**：创作场景下的实时读写不引入加密开销
- **安全责任归属**：磁盘加密由操作系统层解决（见第四节）

> **代码证据**: `db/init.ts:226` — `new Database(dbPath)` 创建标准 `better-sqlite3` 连接，无加密参数；全部 23 个 migration SQL 均为明文 DDL。

### 3.2 不远程同步

CreoNow v0.1 **不**将任何用户数据传输到远程服务器（AI API 调用除外）。

- 无云同步功能
- 无远程备份
- 无遥测数据上传
- AI API 调用仅传输用户显式触发的创作上下文片段，不传输整个数据库

> **代码证据**: 全代码库搜索 `remote.*sync`、`cloud.*backup` 无匹配。网络请求仅限 AI Provider 代理层（`aiProxySettingsService.ts`）。

### 3.3 不提供应用级备份

CreoNow v0.1 **不**内置数据备份或恢复机制。

- 数据持久性完全依赖用户自行备份（如 Time Machine、文件复制等）
- 应用本身不提供版本快照的导出/导入功能（文档版本仅在 SQLite 内部管理）
- 文档级 Markdown / PDF / DOCX / TXT 导出仅用于内容交付，不等同于项目级备份；其输出无法恢复 SQLite 中的版本历史、设置与索引状态
- 如需备份，用户应复制整个 `{userData}/data/` 目录

## 四、安全边界依赖

CreoNow v0.1 的数据安全性依赖以下操作系统级保护措施：

| 保护层 | 机制 | 覆盖范围 |
|--------|-----|---------|
| 全盘加密 | BitLocker（Windows）/ FileVault（macOS）/ LUKS（Linux） | 防止磁盘物理窃取后的数据泄露 |
| 文件系统权限 | NTFS ACL / APFS / ext4 权限 | 防止同机其他用户读取数据目录 |
| 用户账户隔离 | OS 用户账户 + `userData` 目录归属 | 数据库文件仅当前用户可访问 |
| 系统 Keychain | Electron `safeStorage` → Windows DPAPI / macOS Keychain / Linux libsecret | API Key 等敏感凭证的加密存储 |

### ⚠️ 安全边界之外

以下场景**不在** CreoNow v0.1 的安全边界内：

- 未启用全盘加密的设备上，创作内容可通过物理访问磁盘获取
- 共享操作系统账户的场景下，同账户用户可访问数据库文件
- 恶意软件或有 root/admin 权限的程序可读取明文数据库

## 五、已实现的安全措施

### 5.1 API Key 加密存储

API Key 通过 Electron `safeStorage` API 加密后存储在 SQLite `settings` 表中。

**实现路径**：

- **加密入口**: [aiProxySettingsService.ts](../../apps/desktop/main/src/services/ai/aiProxySettingsService.ts) `encryptApiKey()` 函数（L194-L217）
  - 调用 `secretStorage.encryptString(value)` 加密
  - 存储格式：`__safe_storage_v1__:<base64-encoded-ciphertext>`
- **解密入口**: 同文件 `decryptApiKey()` 函数（L223-L255）
  - 检测 `ENCRYPTED_SECRET_PREFIX` 前缀 → `secretStorage.decryptString(buffer)`
- **SecretStorageAdapter 绑定**: [index.ts](../../apps/desktop/main/src/index.ts) L337-L340
  - `safeStorage.isEncryptionAvailable()` / `safeStorage.encryptString()` / `safeStorage.decryptString()`
- **安全约束**: 如 `safeStorage` 不可用，拒绝保存 API Key（返回 `UNSUPPORTED` 错误）

### 5.2 SQLite WAL 模式與安全 Pragma

- `journal_mode = WAL`：防止写入期间数据库损坏
- `foreign_keys = ON`：数据完整性约束
- 推荐 pragma 由 `applyRecommendedPragmas()` 统一设置

## 七、用户建议

为保护创作内容安全，建议用户：

1. **启用全盘加密**：Windows 启用 BitLocker，macOS 启用 FileVault，Linux 启用 LUKS
2. **定期手动备份**：定期复制 `{userData}/data/` 目录至安全位置（U 盘、NAS、加密云盘等）
3. **避免共享账户**：不在多人共用的操作系统账户下存储敏感创作内容
4. **及时更新系统**：保持 OS 安全补丁更新，降低本地提权攻击的风险
5. **妥善管理 API Key**：虽然 CreoNow 使用 safeStorage 加密 API Key，仍建议定期轮换密钥
