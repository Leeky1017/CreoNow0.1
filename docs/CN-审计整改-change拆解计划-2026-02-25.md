# CN 审计整改 OpenSpec Change 拆解计划（2026-02-25）

更新时间：2026-02-25 23:35

## 背景与原则（质量优先）

本计划基于 `docs/CN-补丁堆叠审计报告-整合版-2026-02-25.md`，目标是将审计问题转化为可执行、可验证、可排班的 OpenSpec change backlog，并遵循 CN 治理闭环：

`Issue -> change -> tasks（TDD Mapping）-> RUN_LOG -> PR`。

质量优先原则：
1. 先高风险（P0），再高收益（P1），再中低风险（P2/P3）。
2. 每个 change 必须可测试验收，且能独立回滚。
3. 并行 lane 控制在 2-3 条，避免高冲突文件同时改动。
4. 依赖 change 在进入 Red 前必须完成 Dependency Sync Check 并落盘。
5. 任一 wave 完成后，必须完成主会话审计签字与归档收口，才进入下一 wave。

## 每个 Wave 的进入/退出条件

### Wave 1

进入条件：
1. C1/C2/C3 各自已有 OPEN Issue 与 change 目录。
2. 每个 `tasks.md` 已完成 TDD Mapping，且记录 Red 失败证据计划。
3. `openspec/changes/EXECUTION_ORDER.md` 已登记 3 lane 与依赖。

退出条件：
1. C1/C2/C3 的 P0 验收测试全部通过。
2. RUN_LOG 完成主会话审计签字（`Reviewed-HEAD-SHA==HEAD^`）。
3. PR 合并回 `main`，change 归档完成。

### Wave 2

进入条件：
1. Wave 1 全部归档，主干稳定（required checks 全绿）。
2. C4/C5/C6/C7/C8 的 Dependency Sync Check 已落盘。
3. C10 无依赖，可与 Wave 2 同步启动。

退出条件：
1. 重复定义与类型漂移指标达到目标（重复实现显著归零、强转清零/白名单化）。
2. C10 的 editor save queue 独立模块提取完成并有单测覆盖。
3. 所有 change 的 `tasks.md` 勾选完成并归档。
4. `EXECUTION_ORDER.md` 与实际合并顺序一致。

### Wave 3

进入条件：
1. Wave 2 完成并归档，核心契约稳定。
2. 中低风险 change 的边界与回滚点在 `proposal/tasks` 中明确。

退出条件：
1. 架构 workaround、legacy adapter、风格不一致项完成治理闭环。
2. C15 内存泄漏防护验证通过（watcher 监听器清理 + 全局处理器防重复注册）。
3. 所有 change 完成交付链收口（Issue -> change -> tasks -> RUN_LOG -> PR）。
4. 主干无遗留活跃整改 change。

## 风险与回滚策略

| 风险类型 | 触发信号 | 预防策略 | 回滚策略 |
|---|---|---|---|
| 并发治理引入死锁/吞吐下降（C1/C9） | 压测退化、并发测试波动 | 先加 Red 并发复现测试，再最小改动上锁/串行化 | 新建 rollback issue，回滚到上一个稳定 PR；保留复现测试继续修复 |
| 大规模收敛导致编译/运行断裂（C4/C5） | `tsc` 或集成测试大面积失败 | 分批迁移（按模块批次提交），每批必须可独立通过 | 按批次回退，不整体回滚；保持 `main` 可用 |
| 类型契约漂移（C7/C8） | IPC 返回结构与 renderer 不一致 | 先统一共享类型，再迁移调用方，最后删除兼容层 | 回退 change 并恢复兼容映射，重新执行 Dependency Sync Check |
| 告警噪音过高（C3/C13） | warn 风暴、日志不可读 | 设阈值与节流策略；区分一次性告警与持续告警 | 回滚告警阈值配置或临时降级告警等级 |
| 多 lane 合并冲突 | `EXECUTION_ORDER`/共享文件反复冲突 | 每 wave 固定文件归属；高冲突 change 设为单 lane | 冲突 lane 暂停，新开 issue 做冲突消解 PR |

回滚治理约束：任何回滚仍按 CN 交付链执行，不允许直接在 `main` 手工修补。
