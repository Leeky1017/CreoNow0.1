## 一、护城河判断

Amp 两条 thread 最值得重视的一条判断是：

> CN 的核心竞争力已经不是“会不会调 LLM”，而是“是否拥有一套创作领域特有的结构化理解能力”。

### 1.1 CN 当前真正的差异化资产

| 资产 | 当前形态 | 战略价值 |
|------|----------|----------|
| Knowledge Graph | 角色、实体、关系、状态追踪 | 可以从“写字工具”升级为“世界观管理器” |
| Memory System | 偏好学习、情景记忆、向量召回 | 可以从“AI 会写”升级为“AI 越写越懂你” |
| Context Engine | 分层上下文与预算裁剪 | 可以稳定支撑长篇、复杂、多约束创作 |
| Skill System | 文件化技能定义、路由、执行链 | 可以从内建功能扩展为生态入口 |
| Version / AI Marks | 版本回滚、AI 痕迹标识（当前覆盖文档级快照，多文档/跨项目版本管理尚未闭环） | 可以演化为创作者专属的审计与协作体系 |

### 1.2 为什么这是护城河而不是普通 feature

因为这些能力不是“再加一个按钮”就能复制的，它们需要：

- 持续积累的数据模型
- 稳定的运行时治理
- 写作领域的特定抽象
- 前后端一体化协作

这意味着，一旦把它们转译成好体验，CN 的竞争优势会比单点 feature 持久得多。

## 二、为什么用户还感受不到

### 2.1 当前的主要断裂点

| 能力 | 后端状态 | 用户体感 | 断裂原因 |
|------|----------|----------|----------|
| KG | 已有实体识别与关系结构 | 很多人不知道它重要 | 入口深、显化弱 |
| Memory | 已有偏好学习与注入链路 | 用户很难察觉“AI 变懂我了” | 没有反馈与解释层 |
| Context Engine | 装配与预算治理已成熟 | 用户只看到“AI 有时好、有时一般” | 能力被完全隐藏 |
| Skill System | 架构强、扩展性强 | 用户只把它当技能列表 | 生态潜力尚未产品化 |
| AI Marks | 已有雏形 | 用户不把它当价值点 | 没有被包装成高价值场景 |

### 2.2 搜索与 RAG 发现性：护城河的"入口缺失"

除了上述能力的"界面断裂"，还有一个更基础的问题：**用户写多了之后找不到自己的内容**。

当前缺口：

- 无全局搜索入口（只有编辑器内 `Cmd+F`）。
- KG / Memory 虽然在后端积累了丰富的结构化数据，但用户无法主动检索或利用它来定位旧内容。
- RAG 能力（向量召回）仅服务于 AI 上下文组装，用户自己没有一个"搜索我的创作"的入口。

这意味着：

- 写得越多，反而越难找到旧素材——这对长篇创作者是严重摩擦。
- 护城河积累的数据越丰富，用户感知到的价值却没有同步增长。

推荐方向：

| 阶段 | 能力 | 价值 |
|------|------|------|
| v0.1 | 基础全文搜索（跨文档关键词匹配） | 满足"找到那段话"的最低需求 |
| v0.2 | 语义搜索 / RAG 检索前置 | 让 KG + Memory 的积累直接服务于用户查找 |
| 长期 | 智能素材推荐（"写到这里时你可能需要回顾这段"） | 把搜索从主动行为变成被动辅助 |

### 2.3 当前 UI 的问题不是不支持，而是不翻译

Amp 的批评非常中肯：

- 护城河还停留在后端流转层。
- 用户看到的是一个相对普通的 IDE 外形。
- 真正特别的地方，没有在关键时刻以“眼前一亮”的方式出现。

这意味着 CN 现在更像：

> “背后有很强系统的工具”

而不是：

> “一上手就能感觉到它懂创作的产品”

## 三、护城河显化方案

### 3.1 KG：从后台图谱变成写作现场的角色存在感

最适合 v0.1-v0.2 的路径：

| 方向 | 做法 | 用户感受 |
|------|------|----------|
| 角色卡片自动浮现 | 写到角色名时自动显示角色摘要/关系/最近出场 | “它真的知道我在写谁” |
| KG 前置入口 | 从深层面板入口前移到更自然的位置 | “这个能力不是附属品，而是主功能” |
| 世界观一致性检查 | 自动指出人物设定、关系、时间线冲突 | “它不是续写器，而是世界观守护者” |

### 3.2 Memory：从无形注入变成“越用越懂我”的证据

Amp 指出 Memory 最大的问题不是没有学，而是用户感受不到它在学。

建议路径：

1. 在某次 AI 输出更贴合用户风格时，给出轻提示：
   - “已沿用你最近偏好的叙述节奏”
   - “本次参考了你过去接受过的角色描写风格”
2. 在设置或信息页提供简短可解释视图：
   - 当前学到的偏好摘要
   - 可清空/关闭学习
3. 优先修复“只学 accept 不学 reject”的偏差，不然越解释越容易失信。

### 3.3 Context Engine：从黑箱变成可信赖的创作助手

不需要把技术细节全暴露给用户，但应让用户知道 AI 为什么更稳。

可行方向：

- 在高级模式中展示“本次参考了哪些上下文层”。
- 在 AI 输出质量更高时，用一行人话解释：
  - “已参考角色关系与项目规则”
  - “已结合最近章节摘要”

这样，Context Engine 不再只是隐藏的后端 sophistication，而会变成用户可感知的“AI 不是瞎猜”。

### 3.4 Skill System：从技能列表走向技能生态

Amp 对 Skill System 的判断非常乐观，这点值得保留。

当前优势：

- `SKILL.md` + frontmatter 的文件化定义方式
- builtin / global / project 的三级作用域
- 路由与执行链已具雏形

这意味着未来可以自然走向：

| 方向 | 价值 |
|------|------|
| 用户自定义技能库 | 形成个体工作流资产 |
| 团队/工作室共享技能包 | 形成创作组织能力 |
| Skill Marketplace | 形成社区增长与分发生态 |

### 3.5 AI Marks / Versioning：从版本记录变成出版级审计能力

Amp 提出“AI 编辑痕迹”是出版行业的刚需，这个判断非常值得进入长期路线图。

可发展的方向：

1. 标明哪些段落由 AI 修改、生成、润色。
2. 提供按“修改原因”或“修改角色”组织版本的能力。
3. 向编辑/出版社输出“AI 参与记录”。

这会让 CN 在某些专业场景里，不只是写作工具，而是**可审计的创作系统**。

## 四、V Next 值得下注的方向

### 4.1 世界观一致性检查

这是最像“CN 独家能力”的方向之一。

典型场景：

- 第 3 章角色是蓝眼睛，第 7 章变棕眼睛
- 关系图谱里角色已死亡，后文又正常出场
- 时间线和设定冲突

这类能力一旦做成，会比一般“续写”更有壁垒。

### 4.2 AI 审计轨迹

适合出版、编辑、教育、企业写作等场景。

价值在于：

- 内容来源透明
- 审稿链清楚
- 可区分 AI 润色、AI 生成、人工修改

### 4.3 写作习惯分析

Amp 认为 Analytics 不应只是漂亮页面，而应服务“上瘾感”和自我认知。

可分析维度：

- 写作高峰时段
- AI 使用频率与节省时间估算
- 对话 / 描写 / 叙事占比
- 项目推进节奏

### 4.4 氛围感知上下文层

这是很有 CN 气质的一步。

设想：

- 当前章节偏紧张、温柔、冷峻、轻松
- AI 在生成时自动偏向相应风格

这本质上是把 Context Engine 从“结构感知”继续推进到“氛围感知”。

### 4.5 多章节 / Binder 式创作管理

这虽然不如前几项“酷”，但对长篇写作几乎是刚需。

如果 CN 真要成为“创作者的 IDE”，不能长期停留在弱多文档模型上。

## 五、先后顺序

### 5.1 现在就做的

- 角色卡片自动浮现
- KG / `@` 发现性增强
- AI 修改痕迹的产品化包装
- 让 Memory / Context 的价值有一层轻解释反馈
- 基础全文搜索入口（跨文档关键词检索）

### 5.2 v0.2-v0.3 重点做的

- 世界观一致性检查
- 多章节 / Binder 模型
- 偏好学习负反馈与可解释性
- 写作分析报告
- 语义搜索 / RAG 检索前置

### 5.3 更长期再做的

- Skill Marketplace
- 协作 / 云同步 / 分享链路
- 出版级 AI 审计报告
- 更丰富的主题与个性化工作流系统

### 5.4 顺序背后的原则

1. 先把护城河显化给当前用户。
2. 再把护城河做成难以替代的专业能力。
3. 最后才把护城河外溢成生态和平台。

---

> CN 最大的机会，不是成为“又一个会调模型的写作 App”，而是成为第一个真正把世界观、角色记忆、上下文治理、AI 修改透明度结合起来的创作环境。护城河已经在地底，下一步要做的，是让用户走在地面上时，也能看见山脉的轮廓。
