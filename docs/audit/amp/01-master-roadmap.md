## 一、整理原则

本文件不是原始 thread 的逐段搬运，而是按以下原则重组：

1. 把两条 thread 中重复的判断合并，避免“同一问题说三遍、执行时没人知道先做哪一遍”。
2. 把 Amp 的“发现问题”升级为“问题 -> 根因 -> 动作 -> 验收”的执行格式。
3. 把 thread 中分散的隐患、亮点、机会，统一归到同一优先级体系里。
4. 所有结论都以一个目标收口：**v0.1 发布后，用户能在短时间内感到 CN 不是聊天机器人外接壳，而是真正面向创作者的 IDE**。

## 二、北极星判断

### 2.1 一句话总评

CreoNow 当前处于一种典型的“**内核超前、末梢滞后**”状态：

- 后端能力与 AI 架构，已经超出多数同类写作工具。
- 前端体验、主路径完整性、功能承诺真实性，还没有把这些优势转译成用户可感知的惊艳。

### 2.2 CN 当前最真实的局面

| 维度 | 现状判断 | 结论 |
|------|----------|------|
| AI 能力底盘 | Context Engine + KG + Memory + Skill 已具雏形 | 强 |
| 前端主路径 | 编辑、AI、空状态、引导、发现性存在明显断层 | 弱 |
| 工程规范 | 类型安全、IPC 契约、安全模型都不错 | 强 |
| 防御性工程 | 渲染进程兜底、静默失败可见性、导出真实性不足 | 弱 |
| 护城河显化 | 能力埋在后端，用户很难“看见” | 弱 |
| Windows 首发就绪度 | 可打包，但签名/更新/备份/崩溃可观测性尚未成体系 | 弱 |
| v0.1 风险 | 容易出现“功能看起来很多，但真正上手不惊艳” | 高 |

### 2.3 北极星目标

v0.1 不应追求“功能列表看起来很长”，而应达成以下三件事：

1. 用户打开 CN 后，**30 秒内理解它不是通用聊天工具，而是写作工具**。
2. 用户进入编辑区后，**3 分钟内体验到至少一个值得分享的 Magic Moment**。
3. 用户在保存、切换、AI 生成、导出这些高频动作上，**不会撞上明显失真或静默失败**。
4. 如果以 Windows 为首发主平台，安装、运行、回退与问题定位链路**至少是诚实且可控的**。

## 三、必须保留的强项

Amp 两条 thread 的共识非常清楚：CN 不缺“可继续发展”的内核，缺的是“不要把好内核埋掉”。

### 3.1 不该被改坏的四个核心资产

| 资产 | 当前价值 | 后续要求 |
|------|----------|----------|
| Context Engine | 分层上下文组装、预算裁剪、稳定性强 | 重构时不能破坏层级顺序和 deterministic 行为 |
| Knowledge Graph | 创作工具里少见的结构化角色/关系能力 | 必须从“后端存在”升级到“前端可感” |
| Memory System | 偏好学习、情景记忆、注入链条完整 | 先修学习偏差，再放大其用户价值 |
| Skill System | 文件化技能定义 + 路由 + 执行流水线 | 不要把它只当内部技术设施，应准备生态化出口 |

### 3.2 工程上值得保留的底盘

- Typed IPC + runtime validation：这是 Electron 项目少有的硬底盘，不能为了“快”把契约打散。
- 安全模型：`contextIsolation`、`sandbox`、`safeStorage` 都属于已经做对的部分。
- Skill Scheduler 的并发治理：每会话串行、全局并发上限、取消即释放，是后续扩展 AI 交互的可靠基础。
- 版本快照与三方合并：这使得 CN 后续可以演化为“创作者的 Git”，不要轻易回退到粗暴覆盖写入。当前版本控制覆盖了文档级快照与 AI 编辑痕迹，但多文档/跨项目级别的版本管理尚未形成闭环。

## 四、v0.1 必修项

这里不区分“前端问题”还是“后端问题”，只看它是否会直接伤害首发体验或可信度。

> 从这里开始，不再使用“阻塞 / 不阻塞”的二分法。所有问题都必须被解决，区别只在于：哪些属于首波先做，哪些属于随后收口。

### 4.1 体验侧必修项

| 问题 | 用户看到的现象 | 根因 | 处理方向 |
|------|----------------|------|----------|
| 禅模式不可编辑 | 进入禅模式后只能看，不能写 | `ZenMode` 当前是静态渲染，不是真实编辑器 | 直接挂载 TipTap `EditorContent`，共享 editor 实例 |
| AI 编辑路径过长 | 每次 AI 修改都像离开写作现场去填表 | AI 主路径被设计成右侧面板流程；`Cmd/Ctrl+K` Inline AI **尚未实现**（当前 `Cmd/Ctrl+K` 无绑定） | 新建 Inline AI 输入层 + inline diff，从 0 到 1 实现 |
| Dashboard/空状态没有“写作冲动” | 初次进入像后台系统，不像创作空间 | Hero 区与空状态过于占位化 | 改成内容预览、模板画廊、AI 进度摘要 |
| Onboarding 没有 wow 时刻 | 用户知道“有 AI”，但没体验到 | 第二步只是静态说明 | 在 onboarding 内嵌一个可交互的 AI 演示 |
| Toast 未接入 | 保存/导出/AI/错误无即时反馈 | `Toast.tsx` 完整实现但 `App.tsx` 未挂载 Provider | 挂载 ToastProvider，梳理关键场景接入 |
| Settings 不持久化 | 设置修改重启即丢 | General 页 state 仅组件内，未写入 preferences | 接入 preferences store |

### 4.2 可信度必修项

| 问题 | 现象 | 根因 | 处理方向 |
|------|------|------|----------|
| 自动保存收尾是 `void save(...)` | 切换文档时可能静默丢字 | 异步保存失败无反馈 | 增加失败可见性、重试、状态栏/Toast 提醒 |
| 渲染进程无全局异常兜底 | 某些 async 报错只在控制台出现，用户看不到 | 缺 `unhandledrejection` 全局捕获 | 增加 renderer 全局错误兜底与落盘 |
| PDF/DOCX 导出承诺大于实现 | 用户以为能导出完整排版，实际丢格式/丢图 | 当前为纯文本导出 | v0.1 要么降级为 Markdown-first，要么明确标 Beta |
| Skill Router 关键词路由脆弱 | “不要续写”也可能触发续写 | 只做正向子串匹配，没有否定语境判断 | 增加语义/规则守卫 |
| Windows 首发准备不足 | 可打包不等于可分发、可升级、可定位问题 | 签名/更新/崩溃可观测性缺位 | 将 Win 首发就绪度纳入独立路线 |
| i18n 覆盖不完整 | 切换语言后部分 UI 仍显示硬编码英文 | 存量文案有未走 `t()` 的残留 | v0.1 前完成 i18n key 核查，消灭裸字符串 |
| 全局搜索 / RAG 发现性缺失 | 用户找不到自己写过的段落或 AI 建议 | 尚无全文搜索或语义搜索入口 | v0.1 至少提供基础全文搜索；RAG 发现性列入 v0.2 |
| 本地数据安全未声明 | 创作内容、KG、Memory 均明文存储 | 用户不知道数据保护边界 | 在发布事实表中诚实说明，v0.2 评估加密方案 |
| 备份入口存在但后端无闭环 | 用户以为系统会备份，实际没有可恢复内容 | 后端无调度、无写盘、无恢复 | v0.1 要么实现最小闭环，要么隐藏入口 |
| Judge 入口存在但能力未就绪 | 用户点质量检查后只会得到失败或空结果 | `ensure()` 永远返回 `MODEL_NOT_READY` | 接入远程 LLM，或诚实降级为"基础规则检查" |
| 错误消息直接暴露技术内部 | 用户会看到 `DB_ERROR`、`AI_RATE_LIMITED` 等技术词语 | `errorMessages.ts` 覆盖率不足 7% | 建立统一错误展示层，错误先转译成人话 |
| 文档大小边界未落地 | 超大文档可被直接保存，系统边界失真 | `save` 链路未做 size check | 在保存链路实施 5MB 上限与明确提示 |
| Skill 输出无校验 | AI 异常输出会直接污染正文 | 除 synopsis 外无 output validation | 为高频 skill 增加基础输出校验与兜底提示 |

### 4.3 为什么这些比“继续加功能”更优先

因为它们不是“功能少一点”的问题，而是“**产品对用户说了不该说的大话**”：

- 禅模式说自己是写作模式，却不能写。
- 导出说支持 PDF/DOCX，却无法兑现格式与图片。
- 自动保存说在保护用户内容，却可能静默失败。
- AI 路由说在理解意图，却可能因为一个关键词误会整句话。

这类问题一旦上线，会直接损害信任，而信任是 v0.1 最珍贵的资产。

## 五、三阶段路线图

### 5.1 阶段总览

| 阶段 | 目标 | 关键词 |
|------|------|--------|
| Phase 0 | 先让承诺不说谎 | truthful release |
| Phase 1 | 建立写作主路径的顺手感与惊艳感 | creator flow |
| Phase 2 | 修内功并放大护城河 | hardening + moat |

### 5.2 Phase 0：先把“假能力”收口

目标不是做更多，而是确保 v0.1 的每个入口都真实可信。

优先动作：

1. 禅模式改为真实可编辑。
2. 自动保存失败必须可见、可重试、可追踪。
3. 渲染进程补齐全局异常兜底。
4. 导出能力按真实完成度重新分级：
   - `Markdown` 保持主推。
   - `PDF/DOCX` 若仍不保格式/图片，则改成 `Beta` 或暂时隐藏。
5. Skill Router 至少补齐否定语境守卫，避免明显误判。
6. 明确 Windows 首发边界：签名、更新、备份、崩溃定位四项中哪些已完成、哪些暂不承诺。
7. i18n 存量 key 核查：消灭未走 `t()` 的裸字符串，确保中英文切换不露底。
8. 基础全文搜索入口：用户至少能搜索自己写过的内容。
9. 发布事实表中声明本地数据安全边界（明文存储、无远程同步）。
10. 备份能力做出明确结果：实现最小闭环，或在 v0.1 隐藏入口。
11. Judge 能力做出明确结果：接入可用模型，或诚实降级为基础规则检查。
12. 所有用户可见错误消息统一“说人话”：不再展示技术错误码与内部实现细节。
13. 文档保存链路补齐 5MB 上限，系统边界与 spec 对齐。
14. 高风险 Skill 增加输出校验，不让垃圾输出直接写回正文。

### 5.3 Phase 1：把“创作空间感”做出来

这是 v0.1 真正决定口碑的阶段。

优先动作：

1. Inline AI 快捷路径：把 AI 从右侧面板搬回编辑器语境内。
2. Dashboard Hero 与空状态重做：从占位图标变成内容预览与模板入口。
3. Onboarding 交互式演示：用户在最早阶段就亲手触发一次 AI 能力。
4. `@` 实体引用与快捷键的发现性增强。
5. AI 面板增加简洁模式，减轻信息密度。

### 5.4 Phase 2：一边稳工程，一边把护城河显化

这阶段不是“纯重构”，而是让工程重构服务于差异化。

优先动作：

1. 巨型文件拆分：先拆 `AiPanel.tsx`、`aiService.ts`、`AppShell.tsx`。
2. `AppProviders` 扁平化，减少 11 层 Provider 嵌套。
3. 前端统一日志服务，收口大量 `console.error/warn`。
4. token 估算更贴近中文场景。
5. 偏好学习纳入 `reject` 信号，避免只学“喜欢”、不学“厌恶”。
6. Knowledge Graph 前置为显性能力：
   - 角色卡片自动浮现
   - 图谱入口更靠前
   - 世界观一致性检查立项

### 5.5 GitHub 执行入口（后端治理专题）

本路线图在文档层已经说明了“为什么这些后端问题重要”，但为了避免 GitHub 上重新长出一丛零散 tagged issues，当前已将相关执行入口收束为 4 条母线：

| 母 Issue | 主题 | 在本路线图中的角色 |
|---|---|---|
| [#1007](https://github.com/Leeky1017/CreoNow/issues/1007) | project/session boundary hardening | 承接“首发可信度”里的跨项目隔离、调试权限、来源 ACL 收紧 |
| [#1008](https://github.com/Leeky1017/CreoNow/issues/1008) | preload ipcGateway 与 runtime validation hardening | 承接 payload 估算、structured-clone、timeout / abort、runtime 校验等工程硬化 |
| [#1009](https://github.com/Leeky1017/CreoNow/issues/1009) | shared redaction hardening | 承接凭据与路径脱敏规则库的系统化补真 |
| [#1010](https://github.com/Leeky1017/CreoNow/issues/1010) | CI / discovery / toolchain hardening | 承接门禁真实性、受限环境脚本可运行性、contract / discovery 收口 |

### 5.5.1 为什么不把它们都压成 A0 单点任务

因为这四条线都具有明显的**跨模块、跨阶段**特征：

- #1007 同时横跨 `file/search/context/rag/embedding/constraints`；
- #1008 同时横跨 preload、main、shared 与测试语义；
- #1009 与 UI 脱敏相邻，却不是 UI 脱敏本身；
- #1010 则决定“所有其它门禁到底可信不可信”。

若硬拆成若干个单点 A0 任务，容易再次回到“问题知道很多，执行时一地碎屑”的状态。

因此，本路线图采用：

- **A0/A1/A2 负责阶段性目标**
- **#1007 ~ #1010 负责 GitHub 执行容器**

二者配合，才不至于文档高悬、队列散乱。

### 5.6 Phase 3：V Next 的增长方向

这些不属于首波先做项，但应该现在就纳入路线图，不然会被日常需求淹没。

- 多章节 / Binder 式文档管理
- AI 审计轨迹（出版级透明度）
- 写作习惯分析与反馈
- Skill 社区 / Skill Store
- 氛围感知上下文层
- 协作、同步、分享能力

## 六、依赖关系图

### 6.1 必须先做的底层依赖

| 任务 | 它解锁什么 |
|------|------------|
| 禅模式真实编辑 | v0.1 至少有一个“不会自欺”的写作模式 |
| 渲染进程异常兜底 + 自动保存可见化 | 后续所有体验优化都有了安全护栏 |
| 导出能力重新分级 | 避免发布时承诺失真 |

### 6.2 可以并行推进的两条主线

| 主线 | 包含内容 |
|------|----------|
| 体验主线 | Inline AI、Dashboard 预览、Onboarding 演示、发现性增强 |
| 工程主线 | 拆巨型文件、日志服务、Provider 扁平化、路由与预算修正 |

### 6.3 护城河显化的前提

要把 KG / Memory / Context Engine 变成用户可见优势，至少先满足三件事：

1. 用户能顺畅写作，不被基础流程打断。
2. AI 与编辑器在同一上下文里协作，而非分家。
3. 错误与失败不会被隐藏在控制台里。

否则，再强的后端能力也只会停留在“架构图很漂亮”。

## 七、发布验收板

### 7.1 v0.1 必须通过的体验验收

| 验收项 | 标准 |
|--------|------|
| 禅模式 | 能真实输入、编辑、保存，而非静态展示 |
| AI 编辑 | 选中内容后 4 步内完成一次 AI 修改 |
| 首次体验 | 新用户在 30 秒内触发一次清晰可感的 AI 能力 |
| Dashboard | 能看见项目内容预览或模板，而不是只有占位图标 |
| KG / `@` | 至少有一个显性入口让用户知道它存在 |
| Toast 反馈 | 保存、导出、AI 完成等关键操作有即时 Toast 反馈 |
| 快捷键可发现 | 用户能在 Settings 或帮助中查看完整快捷键列表 |

### 7.2 v0.1 必须通过的可信度验收

| 验收项 | 标准 |
|--------|------|
| 自动保存 | 失败时用户可见，且能重试 |
| 渲染进程异常 | 全局兜底并记录，不允许完全静默 |
| 导出 | 对外宣称的格式能力与真实实现一致 |
| AI 路由 | 否定表达不触发明显反向动作 |
| 队列状态 | 当 AI 请求排队时，用户知道自己在等什么 |
| Win 首发能力边界 | 用户下载安装前就能知道：是否签名、是否支持自动更新、备份是否真实可用 |
| i18n 完整度 | 中英文切换后无裸字符串遗漏 |
| 全文搜索 | 用户能搜到自己写过的内容 |
| 数据安全声明 | 发布说明中诚实交代本地存储安全边界 |
| 无占位 UI | 所有可见按钮/入口要么可用，要么标注 Coming Soon |
| Settings 持久化 | 设置修改重启后保留 |
| 错误消息人话化 | 所有用户可见错误都不出现技术码、栈信息、内部实现名词 |
| Backup 真伪 | 备份要么可真正执行并恢复，要么 v0.1 不展示虚假入口 |
| Judge 真伪 | 质量检查入口与真实能力一致，不允许永远停在 `MODEL_NOT_READY` |
| 文档大小边界 | 超过 5MB 时明确阻止保存并给出可理解提示 |
| Skill 输出安全 | AI 异常输出不会直接污染正文，至少有基础校验与兜底 |

### 7.3 v0.1 必须通过的工程验收

| 验收项 | 标准 |
|--------|------|
| 巨型文件治理 | 至少完成首批高危文件拆分方案，不再继续膨胀 |
| Provider 结构 | 形成统一 `AppProviders` 或等价收口层 |
| 日志 | 前端关键错误不再散落在 `console.*` |
| 测试 | 新增关键用户流测试，而不是只继续堆 existence guard |
| i18n 完整度 | 版本历史、Slash 命令、编辑器硬编码全部走 `t()` |
| 字体一致性 | 实际渲染字体与设计规范一致（加载指定字体或更新规范） |
| Windows 冒烟 | 至少存在一条安装后 Windows 冒烟链路，避免“build 成功、安装即炸” |

---

## 八、明确不做

为了守住节奏，以下事项不应挤进 v0.1 主线：

1. 不为了“看起来现代”而先做云同步、多人协作、账号体系。
2. 不为了“工程完美”而把所有 1000+ 行文件一次性全部拆完。
3. 不为了“功能数”而保留尚未兑现的 PDF/DOCX 强能力宣传。
4. 不把护城河能力继续埋在后端实现里，只写技术文档、不改前端呈现。
5. 不让 v0.1 被“更多 feature”带偏，忽略写作主路径与首发可信度。

---

> 这份主路线图的核心，不是“再多做一点”，而是“先把真正决定口碑的那几件事做对”。CN 不缺聪明的后端，也不缺复杂的系统；现在最缺的是，把这些聪明，变成用户真正能感到的体贴、顺手与惊艳。
