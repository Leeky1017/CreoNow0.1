# CreoNow AMP 实施编排主源


> "名不正，则言不顺；言不顺，则事不成。"——这份文档的任务，不是把任务写满，而是把 AMP 全目录已经形成的判断重新归宗：谁是北极星，谁是证据，谁能排期，谁只能举证。

---

## 一、文档定位

### 1.1 本文是什么

本文是 `docs/audit/amp/` 全目录在执行层的**总编排主源**。

它回答 4 个问题：

1. AMP 全目录中，每份文档在执行体系里扮演什么角色。
2. v0.1 当前真正的主矛盾是什么，哪些任务必须先做，哪些只能辅助并行。
3. Phase 0 / 1 / 2 / 3 的边界、入场条件、完成标志和禁穿透规则是什么。
4. 旧版 `05-implementation-backlog.md` 的任务如何迁移到新的阶段编排之中。

### 1.2 本文不是什么

本文不是：

- `01-master-roadmap.md` 的替代品。北极星仍由 `01` 决定。
- `07 / 08 / 09 / 11` 的专题审计替代品。专题文档提供证据，不自动生成排期。
- `10-phase-0-issue-execution-plan.md` 的替代品。`10` 负责把已确认的 Phase 0 任务转译为 Issue / Change / 审计动作。
- 单个任务的实现设计文档。实现细节仍应回到 OpenSpec、Issue、测试与 PR。

### 1.3 为什么必须重构

旧版 `05` 的主要问题，不在于任务列得不够多，而在于**执行语义失真**：

- 它把专题证据与执行排期混放，导致读者容易把审计结论误读为自动排期。
- 它没有把“信任会破裂”的主线单独抬高，容易让体验优化、工程重构、长期护城河与首发止血相互打断。
- 它低估了 `06 / 09 / 11` 对阶段边界的重写力度，仍保留了旧草案式的乐观推进顺序。
- 它对 `11-frontend-static-code-audit.md` 的纳入方式不清，容易把“前端静态质量问题很多”误读成“前端要先大重构”。
- 它尚未吸收 `12-agent-control-plane-and-incident-reconstruction.md` 的控制面重写，导致“问题为什么会持续再生、如何把规矩编译成硬闸门”缺席于实施编排。

### 1.4 语义优先规则

本文明确采用以下裁决顺序：

1. 以 `docs/audit/amp/` **全目录综合语义**为准。
2. 以 `01-master-roadmap.md` 的北极星、阶段边界、主次关系为准。
3. 以 `06 / 07 / 08 / 09 / 11 / 12` 的新证据修正旧版 `05 / 10` 的乐观假设。
4. **不以更新时间字段作为唯一依据。**若时间较早的文档在综合语义上更高阶，仍然优先。

### 2.1 角色边界声明

- `01` 决定方向，不决定具体 issue 模板。
- `07 / 08 / 09 / 11` 决定“哪里真的在漏水”，但不自动决定“今天先修哪一处”。
- `10` 只在本文确认了 Phase 0 编排之后才有执行合法性。
- `11` 是前端专题中的**高价值证据源**，不是新的总纲，不是新的 backlog，不是前端大重构通行证。
- `12` 定义的是“如何防止同类问题继续长出来”的控制面逻辑；它不替代产品排期，但会重写门禁、审计与实施节奏。

## 三、北极星与总执行原则

### 3.1 北极星

v0.1 当前最优先的，不是“看起来更高级”，而是**不要让用户发现产品在许诺自己尚未兑现的能力**。

当前主矛盾不是审美不够，而是信任会破裂。综合 `01 / 06 / 07 / 08 / 09 / 11`，最危险的裂缝集中在：

1. **假能力或半能力被当成真能力展示。**
	证据：禅模式可见不可写、Inline AI 入口承诺大于实现、Backup / Judge / Export 的能力边界不诚实，分别见 `01`、`07`、`08`。
2. **错误与失败没有被人话接住。**
	证据：错误码和技术信息直出用户表面，详见 `09`。
3. **发布边界不诚实。**
	证据：Windows 首发条件、数据安全边界、备份闭环、更新闭环并未真实成立，详见 `06`、`08`。
4. **前端存在大量静态质量欠债，但其中只有一部分会立即击穿发布信任。**
	证据：`11` 记录了大规模裸字符串、Primitive 绕过、品牌化缺失；其中 i18n 与空壳 UI 属于首发信任问题，品牌化与系统性抽象不属于首发止血主线。

### 3.2 总执行原则

本文据此确立 6 条总原则：

1. **先修真实承诺，再做惊艳体验。**
2. **先收口信任裂缝，再做品牌放大。**
3. **单阶段主线优先于并行冲动。**同一时间只能有一个主阶段牵引全局。
4. **专题审计是证据，不是自动排期。**
5. **发布可信度高于局部优雅。**如果某项改动让架构更好看但不能减少首发风险，应后移。
6. **控制面优先于劝诫。**如果某类问题会持续复发，就必须把它编译成 lint、CI gate、wrapper、截图义务或 fail-closed 审计规则，而不是只在文档里再说一遍。
7. **凡当前大量出现的问题，必须有对应门禁。**未门禁化，不得宣称该问题已治理完成；最多只能说“已识别、待门禁化”。

## 五、当前主线

### 5.1 当前推进阶段

当前唯一主线是 **Phase 0**。

它的核心不是“把所有毛边都磨平”，而是先完成 5 件事：

1. **假能力收口**：禅模式、Inline AI、Backup、Judge、Export、占位 UI。
2. **失败可见**：自动保存、全局错误、错误人话化、错误展示收口。
3. **发布诚实**：发布事实表、Windows 首发边界、数据安全边界。
4. **基础可信输入输出**：i18n 存量清理、文档大小限制、Skill 输出校验、基础搜索入口。
5. **新增失真止血**：把最容易继续再生的问题先堵住——例如裸字符串、Primitive 绕过、raw token 逃逸、fail-open gate 与审计假绿灯。

### 5.2 当前不能打断主线的事项

以下内容即便重要，也不得打断当前 Phase 0 主线：

- Dashboard Hero 重做与品牌氛围放大
- Onboarding 大改
- `AiPanel.tsx` / `aiService.ts` 结构性大拆分
- KG 显化、Memory 解释层、语义搜索前置
- 设计系统抽象补完与 Primitive 体系全面重整

### 5.3 当前可作为并行辅助的事项

只有满足“不改变主线判断、不抢占首发止血注意力”的工作，才允许并行：

- 文档事实核对与发布文案修正
- 决策 issue 的事实核查
- 独立的防御性工程小闭环
- 不依赖上游变更的测试补强
- **不会改变业务行为、但能阻断新增坏味道的控制面加固**（lint / gate / wrapper / review checklist）

- Phase 2 再把 wrapper / schema-first / DAO 边界做实，避免在首发止血阶段被中层大重构夺走注意力。

### 8.1 `07 / 09 / 11` 的分层关系

### 8.2 发布前必修

以下属于前端专题中**发布前必修**的部分：

- `07` 中的假 UI、空壳入口、交互断线、核心路径误导
- `09` 中的错误码直出、技术黑话暴露、统一错误展示层缺失
- `11` 中与首发信任直接相关的 i18n 裸字符串、核心交互组件未走统一机制、明显误导性的表层实现

### 8.3 发布后首批

以下更适合作为 Phase 1 或 Phase 2 的首批项：

- 发现性增强、Dashboard Hero 重构、Onboarding 重做
- Primitive / 组件层统一改造中不影响首发真实度的部分
- 字体、品牌化、视觉层次、局部体感提升

### 8.4 不该过早大重构的内容

以下内容不应借 `11` 的证据规模，在 Phase 0 过早发动：

- 全站组件系统重整
- 以“设计一致性”为名进行大面积表层重写
- 与发布信任无直接关系的品牌化补课
- 只提升代码洁癖、不减少用户风险的前端抽象重构

### 8.5 `11` 的正确落位结论

`11-frontend-static-code-audit.md` 的正确角色是：

- 作为前端专题的**证据库与热力图**；
- 帮助识别哪些问题触及首发信任，哪些只是长期治理；
- 为前端 Agent 提供进入具体模块时的扫描入口；
- **不能**替代 `01` 的阶段判断；
- **不能**替代 `06` 的发布边界；
- **不能**替代 `09` 的错误体验主线；
- **不能**替代 `08` 对后端真伪与能力边界的裁决。

## 十一、执行结论

新的实施编排主源只保留一个中心秩序：

**先修真实承诺，再做惊艳体验；先守住信任，再放大护城河。**

凡与这条秩序相抵触者，即便正确、即便优雅、即便长期有益，也不应在当前主线中抢位。
