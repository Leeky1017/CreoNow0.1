## 一、用户旅程诊断

Amp 两条 thread 共同指出：CN 最大的问题不是“没能力”，而是**旅程断裂**。

### 1.1 当前主路径

`首次打开 -> Onboarding -> Dashboard -> 打开项目 -> 编辑 -> 调 AI -> 保存/版本 -> 导出`

### 1.2 每一段的问题

| 阶段 | 当前问题 | 用户感受 |
|------|----------|----------|
| 首次打开 | Onboarding 第二步是静态说明，不是体验 | “你在跟我讲 AI，而不是让我用 AI” |
| Dashboard | Hero 区与空状态偏占位化 | “像后台工具，不像创作空间” |
| 进入编辑 | 项目切换缺乏仪式感与上下文回顾 | “只是切了个页面，没有进入写作状态” |
| 请求 AI | 右侧面板主导交互，路径长达 10 步左右 | “每用一次 AI 都被打断一次” |
| 使用 KG / `@` / 快捷键 | 能力存在但埋得深 | “我根本不知道这里能这么用” |
| 导出 | 能力描述与真实能力不完全一致 | “我以为它会保留格式，结果没有” |
| 搜索 / 查找 | 无全局搜索入口，只有编辑器内 Cmd+F | “我写了很多，但找不到上周那段话” |

### 1.3 核心结论

当前 CN 的体验问题，本质上不是按钮样式或页面好不好看，而是：

1. **写作流被频繁打断**。
2. **差异化能力没有在恰当时机浮现**。
3. **一些高频入口缺少“真实反馈”与“诚实承诺”**。
4. **i18n 覆盖有缝隙**：部分 UI 文案仍为硬编码英文，切换语言后露底。

## 二、先做：体验生死线

### 2.1 禅模式必须从“观赏品”变成“写作模式”

这一项属于首波先做。

当前问题：

- `ZenMode` 只显示静态段落和假光标。
- 用户进入禅模式后不能真实书写。

对创作产品的伤害：

- 用户会立即产生“这个模式是假的”判断。
- 对“创作者 IDE”的可信度是直接打击。

产品要求：

- 禅模式进入后，用户仍在同一个文档、同一个编辑会话里。
- 保留沉浸式视觉，但不能牺牲真实编辑能力。
- 保存、版本、AI 撤销链路都必须保持一致。

### 2.2 AI 主路径必须回到编辑器现场

Amp 的核心批评非常准确：当前 AI 交互仍然太“面板化”。

当前 10 步左右路径：

1. 选中文本
2. 视线移向右侧
3. 看引用或上下文
4. 选 skill
5. 发起请求
6. 等待流式输出
7. 看 diff
8. 点击应用
9. 再确认
10. 回到编辑器

目标路径应压缩为：

1. `Cmd/Ctrl+K`
2. 输入一句指令
3. 就地预览 diff
4. 回车确认

> **现状勘误**：当前 `Cmd/Ctrl+K` **尚未绑定** Inline AI（`shortcuts.ts` 中 `Cmd/Ctrl+K` 无定义，`Cmd/Ctrl+P` 绑定的是命令面板）。`AiInlineConfirm.tsx` 仅是 AI 结果确认层，不是输入层。因此 Inline AI 是**从 0 到 1 的新建任务**，而非“把面板搬进编辑器”的改造。

这不是“更酷”，而是直接减少心流中断。

### 2.3 Dashboard 与空状态必须承担“唤起写作冲动”的职责

当前 HeroCard 右侧大面积只放置半透明图标，是典型的 UI 占位。

建议重构为三类内容：

| 区域 | 应显示什么 | 价值 |
|------|------------|------|
| Hero 区 | 最近项目的前 2-3 段内容预览 | 让“创作中的作品”成为主角 |
| 新项目入口 | 模板画廊而不是空白按钮 | 降低第一步阻力 |
| 状态摘要 | AI 进度、上次写到哪里、待继续章节 | 提供继续写作的理由 |

### 2.4 Onboarding 必须在 30 秒内交付一次 AI 触感

Amp 对 onboarding 的判断很一致：它解释了 AI，却没有让用户摸到 AI。

应把第二步改成“可交互的微体验”，例如：

- 预置一段短文本，让用户点击一次“继续写”。
- 让 AI 在极短时间内补一小段内容。
- 用一句话解释 KG / Memory 会如何帮助后续创作。

用户不需要在 onboarding 里理解所有系统，只需要在最早时刻感到：

> “这不是一般写作工具，它真能接住我的创作动作。”

## 三、Magic Moment 设计

Amp 两条 thread 提出的 Magic Moment 值得保留，但需要按触发时机重排。

### 3.1 最适合 v0.1 的四个 Magic Moment

| 场景 | 当前基础 | 为什么适合 v0.1 |
|------|----------|-----------------|
| 写到角色名时自动浮现角色卡片 | KG + Entity Matching 已有基础 | 直接把护城河显化为眼前体验 |
| 选中文本后 Inline AI 编辑 | BubbleMenu、流式处理、diff 基础都已存在 | 直接优化主路径 |
| Dashboard 显示真实作品片段 | 编辑器内容和文档状态已有 | 低成本提升“创作空间感” |
| Onboarding 里即时 AI 演示 | `continue-writing` 能力已有 | 最快制造“值得截图传播”的瞬间 |

### 3.2 Magic Moment 的设计原则

1. 不能是纯动画，必须是“**能力显化**”。
2. 不能依赖用户读完说明，必须“**自己跳出来**”。
3. 不能把用户带离写作上下文，必须“**就地发生**”。
4. 不能是重流程，需要“**一击即懂**”。

### 3.3 哪些不是好的 Magic Moment

- 只有炫酷动画，没有实际作用。
- 需要复杂配置后才体验得到。
- 需要进入深层菜单或设置页。
- 只对懂功能的老用户有价值。

## 四、发现性与可理解性

CN 现在的问题不是“功能不存在”，而是“用户不知道它存在”。

### 4.1 被埋没的能力

| 能力 | 当前问题 | 应对方向 |
|------|----------|----------|
| `@` 实体引用 | 几乎没有显性提示 | 在工具栏、空状态、首次使用提示里前置 |
| 快捷键系统 | 已有集中式系统与面板，但日常路径不可见 | 在关键按钮、面板 footer、空状态中露出 |
| Knowledge Graph | 被放在较深的侧边栏/面板逻辑里 | 让图谱成为首页或编辑态中的显性入口 |
| AI 面板高级能力 | 信息密度过高 | 默认简洁模式，高级设置折叠 |
| 全局搜索 | 不存在跨文档搜索入口 | 提供 `Cmd/Ctrl+Shift+F` 级别的全局检索 |

### 4.2 应该怎么教用户，而不是怎么堆功能

- 在用户第一次选中文本时，提示一次 `Cmd/Ctrl+K`。
- 在第一次输入 `@` 时，展示实体补全说明，而不是等待用户碰巧发现。
- 在 Dashboard 上用一句人话解释 KG / Memory / AI 的价值，而不是只展示图标。
- 在 AI 完成一次修改后，顺势解释“你也可以查看 AI 修改痕迹”。

### 4.3 AI 面板的产品策略

AI 面板不应彻底消失，但应从“唯一入口”退为“高级工作台”。

建议分层：

| 层级 | 用途 |
|------|------|
| Inline AI | 高频、轻量、就地修改 |
| 右侧 AI 面板简洁模式 | 长输出、深度创作协作 |
| 右侧 AI 面板高级模式 | 模型/skill/trace/diagnostics 等高级操作 |

## 五、功能缺口判断

这些不是 thread 中提到就立刻都要做，而是要被明确归类，避免后续混成一锅。

### 5.1 v0.1 不该回避的缺口

| 缺口 | 为什么会被立刻问到 | 建议 |
|------|--------------------|------|
| 多章节/多文档管理太弱 | 小说天然不是单文档写作 | 至少进入 v0.2 主计划 |
| AI 历史不完整/不可回看 | 用户会想找回上一次建议 | 先把 chat persistence 列为明确缺口 |
| 导出结果不够忠实 | 创作者高度在意排版与图片 | v0.1 必须诚实降级或补齐 |
| 快捷键与功能发现性弱 | 高价值能力被埋没 | 应纳入当前版本体验优化 |
| 全局搜索 / 内容检索缺失 | 写多了就找不到 | v0.1 至少提供基础全文搜索 |
| i18n 覆盖率不完整 | 切语言后露出未翻译文案 | v0.1 前做一轮 key 核查，消灭裸字符串 |

### 5.2 更适合 v0.2+ 的能力

| 能力 | 原因 |
|------|------|
| 云同步 / 多端协作 | 价值大，但会拖慢 v0.1 节奏 |
| 出版级分享链路 | 需要在内容模型和权限模型上补基础设施 |
| 多主题系统（如 Sepia） | 有价值，但不应先于写作主路径 |
| 拼写/语法检查 | 是好能力，但不应抢在主路径修复前面 |

### 5.3 真正的长线机会

- 世界观一致性检查
- AI 审计轨迹
- 写作状态分析
- Skill 社区生态

这些会在 `04-moats-and-v-next.md` 里展开。

## 六、体验验收清单

### 6.1 首发体验清单

- [ ] 新用户在 onboarding 中亲手触发过一次 AI 能力。
- [ ] Dashboard 不再以占位图标为主，而是以作品/模板/状态为主。
- [ ] 用户能在禅模式中真实写作并保存。
- [ ] 选中文本后存在明显、快速、就地的 AI 修改入口。
- [ ] `@`、KG、快捷键至少各有一个显性提示点。
- [ ] 中英文切换后无裸字符串遗漏。
- [ ] 用户能通过搜索找到自己写过的内容片段。

### 6.2 写作流清单

- [ ] 用户不需要频繁把注意力移到右侧面板才能完成轻量 AI 修改。
- [ ] AI 修改结果在编辑器现场可预览、可接受、可撤销。
- [ ] 项目切换、文档切换、返回 Dashboard 都有清晰上下文。

### 6.3 可信度清单

- [ ] 自动保存失败时，用户能知道、能重试。
- [ ] 导出入口上的能力描述与真实结果一致。
- [ ] 明显的误触发 AI 意图已被削弱。
- [ ] i18n 覆盖率核查完成，关键路径无裸字符串。

---

> 产品体验层面，CN 当前最缺的不是更多页面，而是一个更像“创作空间”的连续体。只要把 AI 拉回编辑器、把 KG 拉到用户眼前、把空状态变成邀请、把承诺变成真实，v0.1 的气质就会从“功能演示”跃迁到“作品生成现场”。
