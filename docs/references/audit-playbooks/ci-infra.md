# CI / 基础设施审计 Playbook

> 适用条件：变更层 WHERE=`infra`（`scripts/**`、`.github/**`、`*.config.*`）
> 审计层级：Tier S 及以上

---

## 必查项（Tier S）

### 1. 幂等性

- [ ] 脚本重复执行是否安全？（不会创建重复资源、不会覆盖已有数据）
- [ ] 如果脚本在中途失败后重新运行，是否能正确恢复？
- [ ] 是否有"已完成则跳过"的守卫逻辑？

### 2. 错误处理

- [ ] 脚本是否设置了 `set -euo pipefail`（或等效的错误退出策略）？
- [ ] 失败后是否有 cleanup 逻辑（临时文件删除、锁释放）？
- [ ] 外部命令（`gh`、`git`、`pnpm`）失败时是否有明确的错误提示？

### 3. 超时与重试

- [ ] 长命令是否有超时设置？
- [ ] 重试逻辑是否有退避策略（exponential backoff）？
- [ ] 是否有最大重试次数？

### 4. CI 工作流

- [ ] Job 是否有 `timeout-minutes` 设置？
- [ ] 条件触发（`if`、`paths`）是否正确——避免不必要的 CI 运行？
- [ ] 是否有 Job 依赖（`needs`）错误导致并行 / 串行不正确？
- [ ] Concurrency 策略是否正确（PR 取消旧运行，main 串行）？

### 5. 权限

- [ ] 脚本是否请求了不必要的权限（如 `GITHUB_TOKEN` 的 `write` 权限）？
- [ ] CI workflow 的 `permissions` 字段是否遵循最小权限原则？
- [ ] 是否有硬编码的 token 或密钥？（禁止）

### 6. 可移植性

- [ ] 是否依赖特定的工具版本（需明确声明）？
- [ ] 是否依赖特定的 OS 特性（如 GNU vs BSD 命令差异）？
- [ ] 环境变量依赖是否有文档或默认值？

### 7. 日志输出

- [ ] 是否有足够的调试输出（printf / echo）帮助排查失败？
- [ ] 日志是否有结构化的阶段标记（如 `=== [1/6] ... ===`）？
- [ ] 敏感信息是否被 mask（`::add-mask::`）？

### 8. 语法与格式

- [ ] Shell 脚本是否通过 `bash -n` 语法检查？
- [ ] 脚本是否有可执行权限（`chmod +x`）？
- [ ] Python 脚本是否通过 `python -m py_compile` 语法检查？

---

## 追加项（Tier D）

### 9. 与交付流程的一致性

- [ ] 脚本变更是否需要同步更新 `AGENTS.md` / `docs/delivery-skill.md` / `scripts/README.md`？
- [ ] CI 工作流的 required checks 是否与 `docs/delivery-skill.md` §二中的定义一致？

### 10. 向后兼容

- [ ] 修改脚本接口（参数、环境变量）是否影响现有调用方？
- [ ] 是否需要更新相关文档中的用法示例？
