# Guard 与 Lint 边界

更新时间：2026-03-07 11:52

## 先讲结论

能用 ESLint 可靠表达的规则，不写 Guard。
Guard 只处理 ESLint 难以覆盖的跨文件、跨层、跨产物一致性问题。

## 什么时候应该写 ESLint

适合 ESLint 的规则：

- JSX 裸字符串
- import 路径约束
- 命名规则
- i18n 调用约束
- 禁止某些 API / 语法 / 原始样式值

原因：

- 反馈更快
- 可在编辑器即时提示
- 粒度更细
- 维护成本更低

## 什么时候才写 Guard

只有当规则满足以下条件之一时，才考虑 Guard：

- 需要跨多个文件联合判断
- 需要读取生成物或目录结构
- 需要验证“契约映射是否成对存在”
- ESLint 无法稳定获得必要上下文

典型候选：

- IPC contract parity
- 原生 binding path 校验
- discovery consistency
- 文档时间戳这类仓库级约束

## 当前仓库的迁移方向

更应迁移到 ESLint 的类型：

- token escape
- i18n hardcode
- import 规范
- 浅层静态字符串匹配类前端约束

更适合保留为 Guard 的类型：

- 跨文件一致性
- 生成物与源文件映射
- 脚本计划与实际发现结果的一致性

## Guard 的写法要求

即使写 Guard，也要保持测试风格统一：

- 使用 vitest
- 使用 `describe/it`
- 每个用例有明确名字
- 输出失败信息要能指导修复

避免：

- 顶层裸块
- 用 `console.log` 代替断言
- 把一组无关规则揉进一个超长脚本

## 评估清单

在新增 Guard 前，先回答：

1. ESLint 真的做不到吗？
2. 这个约束是否跨文件或跨层？
3. 失败后开发者能否快速看懂修复路径？
4. 这个 Guard 会不会在重构时制造大量脆弱误报？

四问中只要前两问不能坚定回答“是”，就先不要写 Guard。
