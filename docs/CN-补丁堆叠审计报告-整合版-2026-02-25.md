## 十二、掩盖架构缺陷的 Workaround

**严重程度：中** | **实例数：4 类（~25 处）**

### 12.1 `void get().refresh()` 模式（~20 处）

在 `kgStore.ts`、`memoryStore.ts`、`projectStore.tsx` 中，几乎每个 mutation 操作后都有 fire-and-forget 的全量重新加载。

**原始缺陷**：store 没有乐观更新机制。
**纠正措施**：实现乐观更新，仅在必要时做后台 sync。

### 12.2 `fireAndForget.ts` 工具函数

`renderer/src/lib/fireAndForget.ts` 将 Promise rejection 转为 console.error。其存在说明代码库中存在大量"不关心结果"的异步操作。

### 12.3 EditorStore 的内联 save queue

`editorStore.tsx:367-465` 在 zustand store 工厂函数内部实现了完整的 save queue（含优先级插入、串行处理、错误恢复）。这 100 行逻辑应该是独立模块。

### 12.4 Context Fetcher 的双重降级

`rulesFetcher.ts`、`retrievedFetcher.ts`、`settingsFetcher.ts` 各自有两层降级。如果 KG 或 Memory 服务长期不可用，上下文层会静默降级，无任何通知。

**纠正措施**：添加降级计数器，连续 N 次降级后发出告警。

## 十四、【新发现】内存泄漏风险

**严重程度：中** | **实例数：2 处**

### 14.1 Watcher 错误监听器未显式移除

**文件**：`services/context/watchService.ts:84-98`

文件监视器的 `watcher.on("error", ...)` 监听器在 watcher 生命周期内从未通过 `.off()` 或 `.removeListener()` 显式移除。虽然 watcher 关闭时监听器会随之释放，但如果 watcher 对象被其他引用持有，监听器会泄漏。

### 14.2 全局异常处理器重复注册风险

**文件**：`globalExceptionHandlers.ts:102-108`

`process.on("uncaughtException", ...)` 和 `process.on("unhandledRejection", ...)` 注册后无防重复机制。如果 `installGlobalExceptionHandlers()` 被多次调用，会累积重复监听器。

**纠正措施**：添加 `once` 标志或在注册前检查是否已注册。

## 十五、【新发现】竞态条件风险

**严重程度：高** | **实例数：3 处**

### 15.1 Episodic Memory Service 并发状态访问（关键）

**文件**：`services/memory/episodicMemoryService.ts:1062-1777`

服务内部维护多个共享可变 Map/Set（`distillingProjects`、`walQueueByProject`、`pendingEpisodeCountByProject` 等），在并发异步操作中无任何同步机制。

**竞态场景**：
```
线程 A: recordEpisode() 检查 distillingProjects.has(projectId) → false
线程 B: scheduleBatchDistillation() 将 projectId 加入 distillingProjects
线程 A: recordEpisode() 继续插入 episode（本应排队）
线程 B: executeDistillation() 运行，遗漏线程 A 的 episode
```

**风险**：丢失更新、蒸馏行为不正确。
**纠正措施**：引入 per-project 互斥锁或串行化队列。

### 15.2 Project Lifecycle 并发切换

**文件**：`services/projects/projectLifecycle.ts:56-240`

`switchProject()` 执行 unbind → persist → bind 序列，但无 per-project 锁。如果两个 `switchProject` 并发调用且涉及重叠的 projectId，unbind/bind 步骤会交错执行。

**纠正措施**：添加 per-project 锁或全局切换锁。

### 15.3 Context Cache 双重计算

**文件**：`services/context/projectScopedCache.ts:18-45`

`getOrComputeString()` 在 cache miss 时调用 `compute()`，但无去重机制。并发请求同一 key 会触发多次计算。

**风险**：浪费计算资源（非数据损坏）。
**纠正措施**：使用 Promise-based 去重（如 `singleflight` 模式）。

## 十六、【新发现】Fire-and-forget 模式

**严重程度：中** | **实例数：~20 处**

代码库中广泛使用 `void promise.catch(...)` 模式（见 §12.1 的 `void get().refresh()` 和 §12.2 的 `fireAndForget.ts`）。

典型示例：
- `index.ts:146,164` — `void Promise.resolve(loadResult).catch(...)` window 加载
- `embeddingQueue.ts:29` — `void Promise.resolve(args.run(task)).catch(...)` 任务执行
- `projectLifecycle.ts:150-164` — `void operationOutcomePromise.then(...)` 错误日志

**风险**：如果 `catch` 回调本身抛出异常，会被静默吞没。无重试机制。
**纠正措施**：对关键操作使用结构化错误上报而非 fire-and-forget。
