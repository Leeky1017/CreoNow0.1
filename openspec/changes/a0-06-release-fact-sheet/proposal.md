# A0-06 发布事实表

- **所属任务簇**: P0-3（能力诚实分级与假功能处置）
- **涉及模块**: workbench
- **前端验收**: 否（纯文档）

---

## Why：为什么必须做

### 1. 用户现象

v0.1 缺少一张诚实的发布事实表——哪些能力真实存在并稳定、哪些处于 Beta 半成品状态、哪些暂不承诺。用户首次打开 CreoNow 看到「禅模式」「AI 润色」「PDF 导出」「知识图谱」「备份」等八九个入口，合理预期是"看到的都能用"。但实际上：禅模式曾长期只读、备份无后端、Judge 长期 `MODEL_NOT_READY`，而导出能力也经历过从“纯文本降级”到“结构化真实交付”的修正过程。用户一兴奋一试一失望，信任链当场断裂——这比功能少更致命。

### 2. 根因

产品没有在发布前建立「能力清单 → 真实分级 → 对外口径」的映射文档。各模块的完成度分散在审计报告、Issue 评论和代码注释中，没有一个面向团队和外部沟通的统一事实源。结果是：

- 市场/文案/首页/App Store 描述无从引用真实能力边界
- 内测用户反馈"这个功能是不是坏了？"——实际上它从未完整实现
- 后续版本无法基于事实表做增量对比（"v0.2 新增了什么"无锚点）

### 3. v0.1 威胁

- **信任透支**：用户试了三个"能用但实际不能用"的功能后，会连真正可用的功能也不再信任
- **沟通失控**：没有统一参照时，不同团队成员对"v0.1 能做什么"各说各话
- **发布合法性**：`01` §5.2 Phase 0 的核心目标是「先让承诺不说谎」——没有事实表等于承诺无锚点

## What Changes：具体做什么

1. **创建 `docs/release/v0.1-fact-sheet.md`**：涵盖 v0.1 全部面向用户功能的真实能力分级
2. **建立三级能力分级体系**：
   - **✅ GA（Generally Available）**：功能完整、经过测试、可对外宣传
   - **⚠️ Beta / 受限**：功能存在但有已知限制，需附带说明
   - **❌ 未实现 / 占位**：UI 入口存在但后端无闭环，不得对外宣传为已有能力
3. **按模块逐项列出真实能力**：每个功能项包含「功能名称」「分级」「用户可见行为」「已知限制」「后续计划」五列
4. **标注数据安全边界**：在事实表独立章节声明「创作内容以明文形式存储在本地文件系统，安全性取决于操作系统级保护」
5. **标注 Windows 首发边界**：签名、自动更新、崩溃可观测性的真实状态

- **下游影响**: A0-07（Windows 首发边界核查）、A0-11（数据安全边界声明）的结论可被事实表引用

## Non-Goals：明确不做什么

1. **不修改任何源代码**——事实表是文档产出，不改变任何模块的运行时行为
2. **不做功能实现或修复**——发现某功能分级为"未实现"时，只记录事实，不在本任务中补实现
3. **不发布到外部渠道**——事实表是内部文档，是对外沟通的素材源，但本任务不包括官网/App Store 文案更新
4. **不替代各模块的独立 spec 文档**——事实表是横切面的能力快照，不取代各模块 spec 的行为定义
5. **不定义 v0.2 / v1.0 的功能规划**——"后续计划"列仅做方向性标注，不承诺具体版本和时间
