# AI Service Specification Delta

更新时间：2026-02-22 19:37

## Change: issue-617-ai-stream-write-guardrails

### Requirement: 流式输出必须批量化并遵守取消优先 [ADDED]

AI Service**必须**在流式输出期间提供可验证的节流/批量化机制，并明确取消竞态规则：

- token/chunk 推送必须批量化（时间窗口/数量阈值），避免事件风暴。
- 取消与完成竞态必须以取消为准，避免“停止生成”无效。

#### Scenario: BE-AIW-S1 chunk batching 按时间窗口/数量阈值合并 [ADDED]

- **假设** LLM 以高频 token/chunk 返回
- **当** AI Service 处理流式响应
- **则** 按时间窗口或最大 batch size 合并输出并批量推送
- **并且** 不会为每个 token 触发一次推送

#### Scenario: BE-AIW-S4 取消与 done 竞态以取消优先 [ADDED]

- **假设** 用户点击“停止生成”与 `done` 事件几乎同时到达
- **当** 系统处理该竞态
- **则** 以取消为准，不应用后续 chunk
- **并且** 最终状态为“已取消”

### Requirement: 写入防护必须提供背压与可回滚的一致性语义 [ADDED]

AI Service**必须**在流式写入相关链路上提供可验证的背压与一致性语义：

- push 背压：在事件风暴下丢弃低优先级输出但保留控制事件必达。
- 写入一致性：一次生成周期的写入必须可整体回滚，避免留下脏数据。

#### Scenario: BE-AIW-S2 事件风暴下丢弃低优先级输出但保留控制事件 [ADDED]

- **假设** 流式输出触发高频 push（事件风暴）
- **当** 触发背压阈值
- **则** 系统可丢弃低优先级输出事件以保护通道
- **并且** 取消/完成等控制事件仍必达

#### Scenario: BE-AIW-S3 abort 触发回滚且不留下部分写入 [ADDED]

- **假设** 流式生成期间存在落盘写入（如内容写入/trace/中间态）
- **当** 用户取消或发生错误导致 abort
- **则** 系统触发回滚，撤销本次生成周期内的写入
- **并且** 不会留下“部分提交”的脏数据
