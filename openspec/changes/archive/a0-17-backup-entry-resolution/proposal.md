## Why：为什么必须做

### 1. 用户现象

用户在 Settings → General 页看到"备份间隔"设置项和"上次备份：2 分钟前"的时间展示，自然会认为系统正在按设定频率自动备份创作内容。但 A0-08 的事实核查已确认：**备份链路完全不存在**——无调度器、无写盘、无恢复入口、时间戳为硬编码。用户以为自己的内容有备份保护，实际完全裸奔。

### 2. 根因

A0-08 事实核查的结论为 v0.1 的 Backup 处置提供了明确的事实基础。问题不在于"是否存在 bug"，而在于**产品需要做出决策**：

- 备份功能从 UI 到后端完全缺失（A0-08 核查结论）
- Settings UI 中的备份设置项在"说谎"——用户以为有保护，实际无保护
- v0.1 的资源有限，是否值得投入工程量实现最小备份闭环，还是诚实隐藏/标注

### 3. v0.1 威胁

- **数据安全信任损害**：备份是用户数据安全的底线。UI 展示备份设置却无实际保护，属于最高级信任违背
- **决策延迟风险**：不做决策，备份入口持续在 UI 中"装作存在"——每多留一天，用户可能多裸奔一天
- **策略一致性**：本决策需与 A0-15（占位 UI 收口）的统一策略协调

## What：这是一个决策任务

本任务基于 A0-08 的事实核查结论，做出 v0.1 对 Backup 入口的最终产品决策。产出物是一份明确的决策结论及实施指令。

### 产出物

1. **决策结论**：从三个方案中选定 v0.1 的处置方式
2. **实施指令**：基于决策结论，给出具体的代码变更要求（哪些组件隐藏、哪些文案修改、哪些 i18n key 增删）
3. **对 A0-06 的影响声明**：决策对发布事实表中备份分级的影响

## Non-Goals：不做什么

1. **不实现完整备份功能**——即使决策结论为"最小闭环"，完整的备份调度器 + 写盘 + 恢复也不在本任务范围内（会拆分为后续任务）
2. **不重新核查备份能力**——A0-08 已完成事实核查，本任务直接消费其结论，不重复劳动
3. **不评估远程备份/云同步方案**——v0.1 仅讨论本地备份能力边界
4. **不修改版本控制模块的快照机制**——版本快照（Snapshot）与备份（Backup）是不同概念，本任务只处理备份
