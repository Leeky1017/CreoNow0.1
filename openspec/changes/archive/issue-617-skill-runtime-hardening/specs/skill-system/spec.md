# Skill System Specification Delta

更新时间：2026-02-22 19:37

## Change: issue-617-skill-runtime-hardening

### Requirement: 技能发现/加载必须懒加载并避免同步全目录扫描 [ADDED]

系统**必须**避免在技能相关操作中反复执行同步目录扫描与同步读文件：

- 技能列表加载必须支持懒加载与缓存（项目级缓存可失效、可回收）。
- 缓存必须与项目切换生命周期对齐：切换时 invalidate，避免跨项目污染。

#### Scenario: BE-SRH-S1 SkillRegistry 项目级缓存避免每次调用扫描目录 [ADDED]

- **假设** 同一项目在短时间内多次触发技能列表/读取操作
- **当** SkillRegistry 提供技能列表
- **则** 使用项目级缓存返回结果而非每次扫描目录
- **并且** 缓存具备容量上限并可在项目切换时失效

#### Scenario: BE-SRH-S2 技能文件读写委托给 DataProcess 异步执行 [ADDED]

- **假设** 某次操作需要读取或写入技能文件
- **当** 系统执行技能文件 I/O
- **则** I/O 以异步方式执行并委托给 DataProcess（或等价 I/O 侧执行环境）
- **并且** 主进程不执行同步 FS 操作以避免阻塞

### Requirement: 调度器必须具备超时回收与取消竞态规则 [ADDED]

技能调度器**必须**在异常与竞态场景下保持可预测：

- 并发槽位必须在 timeout/abort/异常时可回收，即使 completion 丢失也不得永久占用。
- 取消与完成竞态时必须以取消为准，避免用户点击“停止”后仍继续应用输出。

#### Scenario: BE-SRH-S3 completion 丢失时并发槽位仍可回收 [ADDED]

- **假设** 调度器为技能执行占用了并发槽位
- **当** completion 丢失或执行长时间不 settle
- **则** 系统在超时阈值后回收槽位并将任务标记为失败/取消的稳定语义
- **并且** 后续任务可继续执行

#### Scenario: BE-SRH-S4 取消与完成竞态以取消优先 [ADDED]

- **假设** 用户触发取消事件与技能完成事件几乎同时到达
- **当** 系统处理该竞态
- **则** 以取消事件为准，不应用后续输出
- **并且** 最终状态为“已取消”且可验证
