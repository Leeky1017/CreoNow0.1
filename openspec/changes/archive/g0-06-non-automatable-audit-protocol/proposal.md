# G0-06 非自动化问题全覆盖审计协议

- **GitHub Issue**: #1034
- **所属任务簇**: W0-GATE（门禁基础设施）
- **涉及模块**: audit-protocol
- **前端验收**: 否

---

## Why：为什么必须做

### 1. 现状

28 类 AMP 高频问题中，G0-01 至 G0-05 覆盖了 24 类可自动化阻断的问题。剩余 4 类问题**不能用 ESLint/Guard/自动化测试完全覆盖**，因为它们涉及主观判断、架构决策或视觉验收：

| 问题 | 为什么不能自动化 | 当前覆盖 |
|------|-----------------|---------|
| 字体系统未加载/Fallback 不正确 | 需要视觉验收——字体是否正确渲染只能用眼睛看 | 无 |
| 审计只看流程不看产品 | 审计方法论问题，不是代码问题 | 无 |
| 品牌/情绪设计无标准 | 设计审美判断，不能用规则量化 | 无 |
| 全局速率限制缺失 | 架构决策，需要整体设计而非点检 | 无 |

### 2. 根因

- **字体系统**：Web font 是否正确加载、CJK 字符是否使用正确的 fallback 字体——这需要在实际渲染环境中视觉验收。Storybook 可以提供预览，但最终判断需要人眼。
- **审计方法论**：AMP 审计发现一个深层问题——审计 Agent 倾向于检查"代码 diff 是否规范"而非"产品行为是否正确"。这需要更新审计协议，要求审计必须包含产品行为验证步骤。
- **品牌一致性**：CreoNow 定位为"创作者的 Cursor"，需要独特的品牌调性——但目前没有任何标准定义什么是"对的调性"。这需要设计规范文档，而非代码门禁。
- **速率限制**：API 调用频率控制是一个架构级决策——需要在哪些层面、用什么策略限流——不能用 lint 或 guard 解决，需要架构评审。

### 3. 解决路径

这些问题不能自动化，但可以**协议化**——通过 PR review checklist、审计流程更新、文档规范来确保每次 review 都覆盖这些维度。

## What：做什么

### 协议 1: 更新 PR Review Template

**更新 `.github/pull_request_template.md`**，增加"非自动化检查"章节：

```markdown
## 非自动化检查（人工验收，涉及 UI 改动时必填）

- [ ] 字体渲染正确——中英文混排无方块字/fallback 异常
- [ ] 品牌调性一致——新 UI 与现有风格协调，无割裂感
- [ ] 无障碍——键盘可达、屏幕阅读器可理解
- [ ] CJK 场景——中文长文本不截断、不溢出
```

### 协议 2: 更新审计协议（delivery-skill.md §8）

**在审计流程中增加"产品行为验证"步骤**：

- PRE-AUDIT 必须包含：至少 1 条产品行为验证（不能全是 diff review）
- 审计者必须运行应用并验证修改对应的用户场景（至少截图或命令输出）
- FINAL-VERDICT 必须回答："作为用户，修改后的行为是否符合 spec.md 定义？"

### 协议 3: 字体系统验收清单

**新建 `docs/references/font-verification-checklist.md`**：

- Electron 环境中字体加载验证步骤
- CJK 字符渲染验证：中文、日文、韩文代表字符
- Fallback 字体链验证
- Storybook 中的字体预览 story 要求

### 协议 4: 速率限制架构评审

**新建 `openspec/specs/ai-service/rate-limiting-rfc.md`**：

- 记录当前 API 调用链路
- 定义需要限流的端点清单
- 提出候选限流策略（token bucket、sliding window 等）
- 标记为 RFC（Request For Comments）状态，供 A1 阶段决策

---

## Non-Goals：不做什么

1. **不实现速率限制**——仅产出架构 RFC，实现在 A1+ 阶段
2. **不定义品牌视觉规范**——仅建 review checklist 项，完整品牌规范由设计师负责
3. **不强制所有 PR 必须人工视觉验收**——仅要求涉及 UI 改动的 PR
4. **不修改现有审计文档结构**——仅在 delivery-skill.md 中增加条款

---

## 依赖与影响

- **上游依赖**: 无
- **下游受益**: 所有前端 A0 change——PR review 时自动触发非自动化检查项
- **与 AGENTS.md §六 的关系**: 扩充审计协议，增加产品验证维度
