# A0-11 数据安全边界声明

- **GitHub Issue**: #1001
- **所属任务簇**: P0-4（发布边界与数据诚实）
- **涉及模块**: workbench
- **前端验收**: 否（纯文档）

---

## Why：为什么必须做

### 1. 用户现象

用户将小说手稿、角色档案、世界观设定等高价值创作内容托付给 CreoNow。用户的合理预期是"一个专业创作工具会保护我的内容安全"。但实际上——项目文件以 JSON/Markdown 明文存储在用户指定目录，KG 和 Memory 以明文存储在应用数据目录，无加密、无远程同步、无备份引擎。任何拥有同等用户权限的进程都可直接读取全部创作内容。用户对此一无所知，因为产品从未明确告知数据保护边界——不说谎，但也没说真话，"不告而取谓之窃，不言明者亦不诚"。

### 2. 根因

产品在数据安全方面从未建立面向用户的边界声明。当前数据存储架构是：

| 数据类型            | 存储位置        | 格式                 | 加密 |
| ------------------- | --------------- | -------------------- | ---- |
| 项目文件            | 用户选择的目录  | JSON / Markdown 明文 | 无   |
| KG（知识图谱）      | 应用数据目录    | 本地文件             | 无   |
| Memory（向量库）    | 应用数据目录    | 嵌入文件             | 无   |
| 敏感凭证（API Key） | 系统凭证存储    | `safeStorage` 加密   | 有   |
| 编辑器临时缓存      | `.vite-temp` 等 | 临时文件             | 无   |

API Key 通过 `safeStorage` / `SecretStorageAdapter` 加密存储是唯一的安全亮点。但创作内容本身——手稿、角色、知识图谱——全部明文裸奔。此现状本身不是 bug（本地桌面应用的常见选择），但不告知用户就是问题。

### 3. v0.1 威胁

- **安全预期落差**：写作者（尤其是网文作者）对未发表手稿有强烈保护意识。如果用户事后发现创作内容明文存储且从未被告知，信任会瞬间瓦解——"君子坦荡荡"，坦荡的前提是先明言
- **合规风险**：虽然 v0.1 面向个人用户不涉及 GDPR 式强制要求，但缺乏透明度声明在后续版本扩展到团队/企业场景时会成为历史债
- **恶意软件风险**：Windows 平台上恶意软件/间谍软件可直接读取明文创作内容，对商业写作者而言是切实威胁
- **设备丢失场景**：未启用 BitLocker 的 Windows 设备丢失后，硬盘内容可被直接读取

## What Changes：具体做什么

1. **创建 `docs/release/v0.1-data-safety-boundary.md`**：面向用户的数据安全边界声明文档
2. **逐数据类型声明存储方式**：
   - 项目文件（手稿内容）：明文 JSON/Markdown，存储在用户指定目录
   - 知识图谱数据：明文文件，存储在应用数据目录
   - Memory 向量数据：明文嵌入文件，存储在应用数据目录
   - API Key 等敏感凭证：通过 `safeStorage` 加密存储（**唯一已加密项，需明确标注**）
3. **明确声明三个"不"**：
   - **不加密**：创作内容以明文形式存储
   - **不远程同步**：所有数据仅存在于本地
   - **不提供应用级备份**：数据持久性依赖用户自行文件备份（引用 A0-08 结论）
4. **声明安全边界依赖**：安全性取决于操作系统级保护（如 Windows BitLocker、文件系统权限）
5. **标注方向性演进计划**：v0.2 评估 SQLCipher / 加密 FS 对 KG + Memory 的可行性（仅方向性标注，不承诺）
6. **产出可供 A0-06 事实表直接引用的结论**：写入事实表「数据安全边界」章节

## Non-Goals：明确不做什么

1. **不实现数据加密**——本任务只做声明文档，不写加密实现代码
2. **不实现远程同步或云备份**——仅声明"不远程同步"的事实，不做同步方案设计
3. **不在 Settings UI 中新增"数据存储说明"入口**——UI 变更属于后续任务，本任务只产出文档
4. **不评估 GDPR / CCPA 等合规要求**——v0.1 面向个人用户，合规分析不在本范围
5. **不制定加密方案技术选型**——方向性标注"v0.2 评估 SQLCipher"即可，选型分析不在本范围
6. **不修改现有数据存储架构或代码**——声明是只读核查的产出
