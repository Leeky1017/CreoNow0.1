## 变更摘要

创建 v0.1 发布事实表 `docs/release/v0.1-fact-sheet.md`，为每个面向用户的功能建立三级能力分级（GA / Beta / 未实现），作为内部沟通和对外表述的统一事实源。

---

## 变更 Requirement: 发布事实表（新增）

v0.1 **必须**在发布前产出一份结构化的事实表文档 `docs/release/v0.1-fact-sheet.md`，涵盖全部面向用户的功能模块，对每个功能的真实完成度做出诚实分级。

### 能力分级体系

事实表**必须**采用以下三级分级：

| 分级          | 标记 | 含义                       | 对外口径                 |
| ------------- | ---- | -------------------------- | ------------------------ |
| GA            | ✅   | 功能完整闭环，经过测试验证 | 可作为卖点宣传           |
| Beta / 受限   | ⚠️   | 功能存在但有已知限制或降级 | 可提及但必须附带限制说明 |
| 未实现 / 占位 | ❌   | UI 入口存在但后端无闭环    | 不得宣传为已有能力       |

### 文档结构要求

事实表**必须**包含以下章节：

1. **版本信息**：版本号、发布日期、目标平台
2. **能力总览表**：按模块分组，每个功能项包含「功能名称」「分级」「用户可见行为」「已知限制」「后续计划」五列
3. **数据安全边界**：独立章节声明本地存储、明文、无远程同步、无加密的事实
4. **平台支持边界**：Windows 首发状态（签名、自动更新、崩溃可观测性）
5. **分级方法论**：简述分级依据和审计来源，确保可追溯

### 覆盖范围

能力总览表**必须**至少覆盖以下模块的面向用户功能：

- **编辑器**：TipTap 基础编辑、禅模式、大纲、Diff 对比
- **AI 服务**：AI 面板对话、Inline AI、Skill 执行、Judge 质量检查
- **文档管理**：文档 CRUD、文件树、导出（Markdown / PDF / DOCX）
- **项目管理**：项目创建、仪表盘、模板
- **版本控制**：快照、版本历史、版本恢复
- **知识图谱**：实体管理、关系管理、角色系统
- **记忆系统**：写作偏好学习、记忆面板
- **搜索**：全文搜索
- **设置**：通用设置、账户、备份
- **工作台**：命令面板、主题切换、布局

### 分级判据

每个功能的分级**必须**基于以下证据链，禁止凭主观印象判断：

1. 主规范（`openspec/specs/<module>/spec.md`）中定义的行为
2. AMP 审计报告（`docs/audit/amp/`）中记录的缺口
3. 代码现场的实际实现状态

---

### Scenario: S-FACT-1 事实表结构完整性

- **假设** 开发 Agent 完成事实表编写并提交 PR
- **当** 审阅者打开 `docs/release/v0.1-fact-sheet.md`
- **则** 文档包含「版本信息」「能力总览表」「数据安全边界」「平台支持边界」「分级方法论」五个章节
- **并且** 能力总览表按模块分组，每个功能项包含「功能名称」「分级」「用户可见行为」「已知限制」「后续计划」五列
- **并且** 分级标记仅使用 ✅、⚠️、❌ 三种，无其他自创标记
- **并且** 数据安全边界章节明确声明「创作内容以明文形式存储在本地文件系统」

### Scenario: S-FACT-2 能力分级真实性验证

- **假设** 审阅者拿到事实表并逐项核查
- **当** 审阅者对照 AMP 审计报告（`docs/audit/amp/08-backend-module-health-audit.md`）中已确认的缺口
- **则** 备份功能分级为 ❌（后端无调度、无写盘、无恢复）
- **并且** PDF/DOCX 导出分级为 ✅（结构化导出，保留标题/段落/列表等核心语义）
- **并且** Judge 质量检查分级不高于 ⚠️（`ensure()` 返回 `MODEL_NOT_READY`）
- **并且** 禅模式分级反映 A0-01 完成后的真实状态
- **并且** 没有任何已知"后端无闭环"的功能被标记为 ✅

### Scenario: S-FACT-3 事实表可被下游任务引用

- **假设** A0-07（Windows 首发边界核查）和 A0-11（数据安全边界声明）完成后
- **当** 对应结论需要写入事实表
- **则** 事实表的「平台支持边界」和「数据安全边界」章节结构支持直接更新
- **并且** 更新后的内容与 A0-07 / A0-11 的结论保持一致，无矛盾

---

## 约束

1. 事实表是活文档——随各 A0 任务（特别是 A0-04、A0-08、A0-18 的决策结论）完成后需同步更新
2. 分级必须基于证据链，禁止「据代码观察」式空口判断
3. ❌ 分级的功能必须在「已知限制」列说明 UI 入口的当前处置方式（隐藏 / 标注 Coming soon / 保持现状）
4. 事实表不作为对外发布的最终文案——它是内部事实源，对外文案由产品团队基于此表编写
