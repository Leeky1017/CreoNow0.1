# Delta Spec: ai-service — Judge 能力决策

- **Parent Change**: `a0-18-judge-capability-resolution`
- **Base Spec**: `openspec/specs/ai-service/spec.md`
- **GitHub Issue**: #997
- **类型**: 决策框架（Decision Framework）——本文档定义决策过程的结构化要求

---

## 变更摘要

QualityPanel 的质量检查功能因 `JudgeService.ensure()` 永远返回 `MODEL_NOT_READY` 而完全不可用。本 spec 定义 v0.1 对 Judge 能力的产品决策框架：接入远程 LLM、降级为基础规则检查、或隐藏入口。

---

## 决策框架

### 一、必答问题（Must-Answer Questions）

以下问题**必须**在决策文档中逐一回答，每个答案**必须**附代码证据：

| 编号 | 问题                                                                        | 预期证据类型                   |
| ---- | --------------------------------------------------------------------------- | ------------------------------ |
| Q1   | QualityPanel 质量检查按钮是否已渲染？绑定了什么事件？                       | 组件文件路径 + 代码片段        |
| Q2   | 质量检查的 IPC 调用链路是什么？（UI → IPC → Service 完整路径）              | IPC 注册代码 + Service 方法    |
| Q3   | `JudgeService.ensure()` 为什么返回 `MODEL_NOT_READY`？检查了什么条件？      | Service 源码中 `ensure()` 方法 |
| Q4   | LLM Proxy 配置链路当前就绪状态如何？API Key 存储、provider 选择是否已可用？ | LLM 配置相关代码搜索           |
| Q5   | 若接入 LLM，Judge 需要什么 prompt？当前是否已定义 prompt template？         | Judge prompt 相关代码          |
| Q6   | QualityPanel 中除质量检查外，还有哪些功能？是否可独立使用？                 | QualityPanel 组件代码          |
| Q7   | ai-service spec 中是否定义了 Judge 的行为规范？定义了什么？                 | spec.md 相关段落               |
| Q8   | 降级为"基础规则检查"需要哪些规则？可否不依赖 LLM 实现？                     | 创作规则引擎可行性评估         |

### 二、方案对比（Option Comparison）

决策文档**必须**对以下三个方案进行五维度结构化对比：

#### 方案 A：v0.1 接入远程 LLM，让 Judge 可用

| 维度                | 评估要求                                                                                   |
| ------------------- | ------------------------------------------------------------------------------------------ |
| **用户影响**        | 用户获得 AI 质量反馈能力——产品核心差异化价值                                               |
| **实现成本**        | 量化：需打通 LLM Proxy → Judge 的调用链路、需编写 prompt template、需处理 LLM 不可用的降级 |
| **风险**            | 依赖用户配置 API Key——未配置时 Judge 仍不可用；LLM 调用延迟和成本                          |
| **可逆性**          | 中——已实现的 Judge 逻辑需长期维护                                                          |
| **对 A0-06 的影响** | 事实表中 Judge 标记为"⚠️ 需要 LLM 配置"，注明"用户未配置 API Key 时不可用"                 |

#### 方案 B：v0.1 降级为"基础规则检查"（不依赖 LLM）

| 维度                | 评估要求                                                             |
| ------------------- | -------------------------------------------------------------------- |
| **用户影响**        | 用户获得基础质量反馈（字数统计、段落长度、标点检查等），但非 AI 级别 |
| **实现成本**        | 量化：需实现规则引擎、需定义规则集、需修改 UI 展示                   |
| **风险**            | 用户可能期望"AI 质量检查"但获得"规则检查"——名实不符风险              |
| **可逆性**          | 高——规则引擎可在 v0.2 中被 AI Judge 替代或增强                       |
| **对 A0-06 的影响** | 事实表中 Judge 标记为"⚠️ 基础规则检查"，注明"非 AI 评估"             |

#### 方案 C：v0.1 隐藏 QualityPanel 质量检查入口

| 维度                | 评估要求                                                  |
| ------------------- | --------------------------------------------------------- |
| **用户影响**        | 用户不再看到无法兑现的质量检查承诺                        |
| **实现成本**        | 低——隐藏按钮或整个 QualityPanel（视 Q6 答案决定）         |
| **风险**            | 产品核心差异化能力暂时缺席——"创作者的 Cursor"没有质量反馈 |
| **可逆性**          | 高——后续版本可随时恢复入口                                |
| **对 A0-06 的影响** | 事实表中 Judge 标记为"❌ v0.1 未提供"                     |

### 三、决策门槛（Decision Threshold）

Owner 做出最终决策前，以下条件**必须**全部满足：

1. **Q1-Q8 全部已回答**：每个问题有明确结论和证据，无"待确认"项
2. **三方案对比完成**：每个方案的五个维度评估均已填写
3. **LLM Proxy 就绪度已确认**：明确回答"当前 LLM 配置链路能否支撑 Judge 调用"
4. **与 A0-15 策略一致性已确认**：若选择隐藏方案，处置规则与占位 UI 统一策略一致

## 约束

1. 本任务**只做事实核查和决策框架输出**——不写实现代码
2. 核查过程**只读代码、不修改代码**——所有验证通过 grep、搜索、阅读完成
3. 决策建议**必须基于事实**——禁止"根据经验判断"或"业界通常做法"等无证据论述
4. 最终决策权在 Owner——Agent 输出建议，Owner 拍板
5. 若选择方案 B（降级），"基础规则检查"的规则集定义为**后续任务**——本任务仅评估可行性，不设计具体规则
