## 变更 Requirement: Windows 首发边界核查文档（新增）

v0.1 **必须**在发布前产出一份结构化的 Windows 首发就绪度核查文档 `docs/release/v0.1-windows-boundary.md`，逐项核查并诚实记录四大维度的真实状态。

### 核查维度

核查文档**必须**覆盖以下四大维度：

| 维度         | 核查要点                                                  | 证据来源                             |
| ------------ | --------------------------------------------------------- | ------------------------------------ |
| 代码签名     | 证书配置、CI 集成、SmartScreen 处理                       | `electron-builder.json`、CI workflow |
| 自动更新     | electron-updater 接入、publish provider、更新通道         | `electron-builder.json`、主进程代码  |
| 备份能力     | 调度器、写盘服务、恢复入口                                | A0-08 核查结论、`06` §4.1            |
| 崩溃可观测性 | globalExceptionHandlers 覆盖、renderer 异常捕获、遥测管线 | 主进程 / renderer 代码、`06` §4.2    |

### 文档结构要求

核查文档**必须**包含以下章节：

1. **核查概述**：核查目标、核查日期、核查方法
2. **维度核查详情**：四大维度逐项展开，每项包含「当前状态」「证据」「就绪度标记」「v0.1 处置建议」
3. **就绪度总表**：四维度的一览表，使用 ✅ / ⚠️ / ❌ 标记
4. **结论与建议**：总结性结论，哪些可承诺、哪些需诚实声明暂不承诺

### 就绪度标记体系

每个维度的就绪度**必须**使用以下三级标记：

| 标记        | 含义                               |
| ----------- | ---------------------------------- |
| ✅ 已就位   | 已实现且可工作，可对外承诺         |
| ⚠️ 部分就位 | 框架或骨架存在但未闭环，需附带说明 |
| ❌ 未就位   | 完全缺失或仅有占位，不得对外承诺   |

### 证据要求

每个维度的核查结论**必须**附带以下证据之一：

- 代码路径和具体实现的引用（文件名 + 行号范围）
- 审计文档的引用（`06` 章节号）
- 搜索命令和结果（`grep -rn` 等输出摘要）

禁止"据观察"式空口判断。

### Scenario: S-WIN-1 核查文档结构完整性

- **GIVEN** 开发 Agent 完成 Windows 首发边界核查并提交 PR
- **WHEN** 审阅者打开 `docs/release/v0.1-windows-boundary.md`
- **THEN** 文档包含「核查概述」「维度核查详情」「就绪度总表」「结论与建议」四个章节
- **AND** 维度核查详情覆盖代码签名、自动更新、备份能力、崩溃可观测性四项
- **AND** 每个维度包含「当前状态」「证据」「就绪度标记」「v0.1 处置建议」四个子项
- **AND** 就绪度标记仅使用 ✅、⚠️、❌ 三种

### Scenario: S-WIN-2 核查结论与代码现场一致

- **GIVEN** 审阅者拿到核查文档并逐项核查
- **WHEN** 审阅者对照 `electron-builder.json` 和主进程/renderer 代码验证
- **THEN** 若 `electron-builder.json` 无 `publish` / `win.sign` 配置，自动更新和代码签名维度就绪度不高于 ⚠️
- **AND** 若搜索 `autoUpdater` / `electron-updater` 无功能性调用，自动更新标记为 ❌
- **AND** 若 renderer 中 `fireAndForget.ts` 仍有 `TODO` 标注未接 telemetry，崩溃可观测性不标记为 ✅
- **AND** 备份维度引用 A0-08 结论，不自行推测

### Scenario: S-WIN-3 核查结论可被事实表引用

- **GIVEN** A0-06（发布事实表）需要填写「平台支持边界」章节
- **WHEN** A0-06 执行 Agent 引用本任务产出
- **THEN** 就绪度总表的四维度状态可直接迁移到事实表
- **AND** 每个维度的处置建议足够明确，无需 A0-06 执行 Agent 额外调研

---

## 约束

1. 核查是只读操作——不修改任何代码、配置或构建文件
2. 核查结论必须基于当前代码现场，不得基于"计划要做"来标高就绪度
3. 备份维度必须引用 A0-08 的正式结论，不重复核查
4. 核查文档不包含实现方案——"缺什么"归核查，"怎么补"归后续任务
