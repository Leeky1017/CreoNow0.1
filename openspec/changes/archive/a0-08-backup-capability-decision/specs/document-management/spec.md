# Delta Spec: document-management — 备份能力真伪核查（决策框架）

- **Parent Change**: `a0-08-backup-capability-decision`
- **Base Spec**: `openspec/specs/document-management/spec.md`
- **GitHub Issue**: #1035
- **类型**: 决策框架（Decision Framework）——本文档不定义行为变更，而是定义决策过程的结构化要求

---

## 变更摘要

对备份功能的 UI 承诺与后端实现进行全面事实核查，输出结构化决策文档，为 A0-17 的产品决策提供事实依据。本 spec 不修改任何运行时行为，仅定义核查过程和产出物的质量要求。

---

## 决策框架

### 一、必答问题（Must-Answer Questions）

以下问题**必须**在决策文档中逐一回答，每个答案**必须**附代码证据（文件路径 + 行号或搜索命令）：

| 编号 | 问题                                                                          | 预期证据类型                     |
| ---- | ----------------------------------------------------------------------------- | -------------------------------- |
| Q1   | Settings UI 中 `backupInterval` 控件是否已渲染？用户能否交互？                | 组件文件路径 + 截图或代码片段    |
| Q2   | `backupInterval` 用户选择的值是否被持久化？存储在哪里？                       | Store / PreferenceStore 代码路径 |
| Q3   | 是否存在 `backupService` 或任何名称包含 `backup` 的 Service 类？              | `grep -r` 搜索结果               |
| Q4   | 是否存在备份相关的 IPC handler（如 `backup:start`、`backup:restore`）？       | IPC 注册代码搜索结果             |
| Q5   | 是否存在定时调度逻辑（`setInterval`、`cron`、`electron-scheduler`）执行备份？ | 全局搜索结果                     |
| Q6   | 备份数据写入何处？文件系统独立目录？SQLite 附加表？                           | 写盘逻辑代码或「不存在」声明     |
| Q7   | 是否存在备份恢复入口（UI 按钮或 IPC handler）？                               | UI 组件 / IPC 搜索结果           |
| Q8   | "上次备份：X 分钟前"的时间戳数据源是什么？是真实计算还是硬编码？              | 时间戳来源代码                   |
| Q9   | i18n 文案中有多少条备份相关的 key？内容是否暗示备份已在工作？                 | locale 文件 `grep` 结果          |
| Q10  | document-management spec 中是否定义了备份行为？如果有，定义了什么？           | spec.md 相关段落                 |

### 二、方案对比（Option Comparison）

决策文档**必须**对以下三个方案进行结构化对比：

#### 方案 A：v0.1 隐藏备份入口

| 维度                | 评估                                                     |
| ------------------- | -------------------------------------------------------- |
| **用户影响**        | 用户不再看到无法兑现的备份承诺，避免信任违背             |
| **实现成本**        | 低——隐藏 UI 控件 + 移除相关 i18n 文案                    |
| **风险**            | 若用户已依赖备份设置的心理预期，移除可能引起"功能倒退"感 |
| **可逆性**          | 高——后续版本可随时恢复入口                               |
| **对 A0-06 的影响** | 事实表中备份标记为 ❌，注明"v0.1 未提供"                 |

#### 方案 B：v0.1 实现最小备份闭环

| 维度                | 评估                                          |
| ------------------- | --------------------------------------------- |
| **用户影响**        | 用户获得真实的基础备份保护                    |
| **实现成本**        | 中高——需要调度器、写盘、恢复入口、测试        |
| **风险**            | 在 Phase 0 排期中挤占其他更高优先级任务的资源 |
| **可逆性**          | 中——已实现的备份逻辑需要长期维护              |
| **对 A0-06 的影响** | 事实表中备份标记为 ⚠️，注明最小闭环的能力边界 |

#### 方案 C：v0.1 保留入口但标注"Coming Soon"

| 维度                | 评估                                              |
| ------------------- | ------------------------------------------------- |
| **用户影响**        | 用户知道功能存在但尚未就绪，期望管理清晰          |
| **实现成本**        | 低——添加 disabled 状态 + 提示文案                 |
| **风险**            | 如果同类"Coming Soon"过多，产品整体显得半成品     |
| **可逆性**          | 高                                                |
| **对 A0-06 的影响** | 事实表中备份标记为 ❌，注明"入口保留，标注未就绪" |

### 三、决策门槛（Decision Threshold）

Owner 做出最终决策前，以下条件**必须**全部满足：

1. **Q1-Q10 全部已回答**：每个问题有明确结论和证据，无"待确认"项
2. **三方案对比完成**：每个方案的五个维度（用户影响、实现成本、风险、可逆性、对 A0-06 影响）均已填写
3. **与 A0-15（占位 UI 收口）的一致性已确认**：备份入口的处置方式与占位 UI 统一策略不冲突
4. **A0-17 可直接消费**：决策文档的结论格式满足 A0-17 的输入要求——即 A0-17 的执行 Agent 可直接基于本文档行动，无需额外调研

- **当** Owner 检查决策门槛清单

- **并且** 三方案对比已完成
- **并且** 与 A0-15 占位 UI 策略的一致性已确认
- **并且** 文档结论可被 A0-17 直接消费

## 约束

1. 本任务**只查事实、不做实现承诺**——产出物是决策框架，不是实现方案
2. 核查过程**只读代码、不修改代码**——所有验证通过 grep、搜索、阅读完成
3. 决策建议**必须基于事实**——禁止"根据经验判断"或"业界通常做法"等无证据论述
4. 最终决策权在 Owner——Agent 输出建议，Owner 拍板，A0-17 执行
