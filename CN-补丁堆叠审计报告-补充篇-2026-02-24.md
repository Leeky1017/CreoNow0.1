## 二、静默错误抑制

### 分类统计

在全代码库中发现 **50+ 个 `} catch {` 块**（无变量捕获的空 catch）。按行为分为三类：

### 2.1 合理的静默抑制（~20 处）— 不需要修复

- **JSON 解析降级**：`JSON.parse` 失败返回 null/默认值（`aiProxySettingsService.ts:112`、`kgCoreService.ts:293`、`memoryService.ts:315` 等）
- **URL 解析容错**：`new URL()` 失败返回 null（`browserWindowSecurity.ts:33`、`ipcAcl.ts:54`）
- **文件存在检查**：`fs.statSync` 失败返回 false（`templateService.ts:58`、`index.ts:95`）

### 2.2 问题性静默抑制（~15 处）— 需要修复

| 文件 | 行号 | 原始错误 | 补丁行为 | 风险 |
|------|------|----------|----------|------|
| `index.ts:466` | `catch { // Swallow }` | 日志写入失败 | 完全吞没，启动失败时无任何诊断 | **高** |
| `fakeAiServer.ts:309` | `catch { // Ignore }` | `res.end()` 失败 | 吞没网络错误 | 低 |

| `AiPanel.tsx:497` | `catch { return; }` | localStorage 读取失败 | 静默丢弃，用户设置丢失无提示 | 中 |
| `AiPanel.tsx:532` | `catch { return; }` | localStorage 解析失败 | 同上 | 中 |
| `AiPanel.tsx:543` | `catch { return; }` | localStorage 写入失败 | 同上 | 中 |
| `AiPanel.tsx:698` | `catch { // ignored }` | judge 评估失败 | 静默丢弃评估结果 | 中 |

- 2.3 中的降级项：在 catch 块中添加 `logger.warn()` 调用

## 四、硬编码值作为快速修复

### 4.1 `max_tokens: 256` 硬编码在 Anthropic API 调用中

| 文件 | 行号 | 上下文 |
|------|------|--------|
| `services/ai/aiService.ts:718` | `max_tokens: 256` | 非流式 Anthropic 请求 |
| `services/ai/aiService.ts:879` | `max_tokens: 256` | 流式 Anthropic 请求 |

## 五、掩盖架构缺陷的 Workaround

### 5.1 `void get().refresh()` 模式（~20 处）

在 `kgStore.ts`、`memoryStore.ts`、`projectStore.tsx` 中，几乎每个 mutation 操作后都有 `void get().refresh()`——发起一个 fire-and-forget 的全量重新加载。

**原始缺陷**：store 没有乐观更新机制，mutation 后不知道本地状态应该如何变化。
**补丁**：用全量 refresh 替代精确的状态更新。
**风险**：中 — 每次操作触发额外的 IPC 往返；refresh 失败时静默丢弃错误。
**纠正措施**：实现乐观更新（mutation 成功后直接更新本地状态），仅在必要时做后台 sync。

### 5.2 `fireAndForget.ts` 工具函数

`renderer/src/lib/fireAndForget.ts` 提供了一个将 Promise rejection 转为 console.error 的工具。这本身是防御性编程，但它的存在说明代码库中存在大量"不关心结果"的异步操作。

### 5.3 EditorStore 的内联 save queue

`editorStore.tsx:367-465` 在 zustand store 工厂函数内部实现了一个完整的 save queue（含优先级插入、串行处理、错误恢复）。这 100 行逻辑应该是独立模块，而非嵌入在 store 定义中。这也是 `eslint-disable max-lines-per-function` 的根因。

**纠正措施**：提取为 `createSaveQueue()` 独立模块。

### 5.4 Context Fetcher 的双重降级掩盖 KG/Memory 不可用

`rulesFetcher.ts`、`retrievedFetcher.ts`、`settingsFetcher.ts` 各自有**两层**降级：内层 `!res.ok` → 返回 warning；外层 `catch` → 返回 warning。如果 KG 或 Memory 服务长期不可用，上下文层会静默降级，用户和开发者都不会收到任何通知。

**纠正措施**：添加降级计数器或持久化日志，连续 N 次降级后发出告警。

## 七、多贡献者导致的不一致模式

### 7.1 Store Provider：JSX vs React.createElement

如 §3.4 所述，5 个 `.ts` store 用 `React.createElement`，2 个 `.tsx` store 用 JSX。

**纠正措施**：统一为 `.tsx` + JSX。

### 7.2 错误消息语言混用

后端 IPC 层：部分用中文（`runtime-validation.ts`、`providerResolver.ts`），部分用英文（大多数服务文件）。前端 UI：中文为主但有英文混入（如 `"projectId is required"`）。

**纠正措施**：后端统一使用英文 error code + message，前端负责翻译。

### 7.3 `ipcError` 函数签名微变体

30 个文件中的 `ipcError` 大多数签名为 `(code, message, details?)`，但：
- `ipc/stats.ts:26` 和 `ipc/constraints.ts:223`：`code` 类型是 `IpcError["code"]` 而非 `IpcErrorCode`
- `services/judge/judgeService.ts:28` 和 `services/ai/judgeQualityService.ts:42`：只有 `(code, message)`，无 `details` 参数
- `services/projects/projectService.ts:197`：多了 `traceId` 和 `retryable` 选项

这些细微差异说明不同时期添加的服务各自演化了自己的变体。
